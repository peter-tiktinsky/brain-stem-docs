Sessions · tidying up

Session close

Ending a session has an always-on side that happens whether you ask or not — your session leaves the registry, and on a clean exit the vault gets an automatic integrity pass — and an explicit close you can run to do that tidying on demand and get a backup offer. Either way a whole checklist of housekeeping runs in the right order, so you don't have to remember a dozen steps yourself.

Always-on: session-deregister.sh. Explicit: /librarian session-close.

The always-on half — auto-close and reconcile

A session leaves the registry without you doing anything — and there are two cases, because a session doesn't always end cleanly:

  • A clean end. Closing the window fires the SessionEnd hook session-deregister.sh, which marks your registry row closed and, if a peer left work pending, spawns the reconciler in the background — detached and non-blocking.
  • An abandoned or crashed session. If the process just vanishes and SessionEnd never fires, your row is left behind as a ghost — and it is auto-closed by the reconciler. A row is reaped once it looks gone — its process is dead or its heartbeat has gone stale (older than ~30 minutes) — but a fresh heartbeat is a hard keep that overrides a pid which merely looks dead, so a live session that is still heartbeating is never wrongly reaped.

This reconciliation is effectively always-on: it runs at session lifecycle moments — a new session reaps on start, a closing session spawns it when a peer is pending — reaping the stale, closed, and pending rows under a lock, clearing the pending flag, and stamping the time. It is idempotent (running it twice changes nothing). Its job is to keep peer-awareness honest so the shared "whiteboard" never fills with ghosts. Note what it is: registry cleanup — the automatic close of a session's coordination footprint — not the thorough housekeeping sweep itself.

That registry cleanup is not the only thing that happens on its own. When a session ends cleanly without a formal close, brain-stem also spawns the integrity subset of the close automatically, detached in the background — so the thorough housekeeping mostly runs without you asking: it re-checks conventions, refreshes the plan and writer indexes, re-syncs master plans, flags drift, and refreshes the project binder and work-surface. It is structurally commit-free (it never commits, never offers a backup, and never rewrites your authored content), it auto-detects its scope, and it stands down if you already ran a formal close. If a session crashes before it can run, the next session's start-up runs it as a backstop. So a formal /librarian session-close is mostly about doing this now, on demand — plus the one thing the automatic pass deliberately won't do: offer to back up.

The explicit half — /librarian session-close

When you declare you're done, the close capability first auto-detects its scope from the registry, then chains a fixed sequence of cleanup chores:

Scope: solo

You're the only session

Run the full cleanup, including the shared reconciliation.

Scope: scoped

Peers still active

Do only your own part; defer the shared sweep to whoever closes last.

Scope: reconciler

You're the last one out

Run your part and the deferred shared sweep on everyone's behalf.

The binder and work-surface refreshes are scoped to the project you're closing in — they regenerate exactly the context the next session in that project loads at start.

Advisory, never wedged

The whole flow is advisory: a chore that fails is logged and the flow continues, and the close always exits cleanly. It never blocks you and never leaves a session stuck. The drift check is read-only; the cascade-rename is a proposal, not an auto-apply. The close also never commits anything itself — a manual close just ends by offering to run /librarian backup if you want to commit the working state.

The receipt — the session-close log

Every close writes one aggregated session-close log — the librarian's notes on what the close actually did, with each chore marked ok / skip / error. It's the always-there receipt: a record exists even when the cleanup ran quietly in the background. It lives in machine-local state, not the vault — machine exhaust stays out of your indexed knowledge.

How the two work in concert

The two halves are designed as a floor and a sweep. The always-on deregister is the floor: every session, no matter how it ends, leaves the registry honest and triggers background reconciliation when a peer needs it — so concurrent sessions never corrupt the shared list. The thorough integrity sweep runs on top — automatically on a clean exit, or on demand when you close formally. The coordinating rule is "last session out does the shared cleanup": if peers are still active when you formally close, your close does only your part and defers the shared reconciliation to whoever closes last — which is why two closing sessions never trample each other or run the global cleanup twice. Because the reconciler is idempotent, that deferral is always safe.

What if you never formally close?

Less is lost than you'd think. Closing the window cleanly already triggers the automatic integrity pass above, so the thorough housekeeping mostly runs on its own. What a formal close adds is doing it now and on your terms — plus the one chore that is never automatic: the backup offer.

Still happens automatically

Your session is marked closed in the registry; if a peer left work pending, the reconciler runs in the background; the session's episodic-memory row is written; and — on a clean exit — the integrity pass re-checks conventions, refreshes the indexes, re-syncs masters, flags drift, and refreshes the project binder and work-surface, writing its own receipt. Your actual work — every file you wrote — stays in the vault and the plan tree, untouched.

What you actually give up

Two things. First, backup is never automatic — only a manual close offers /librarian backup; nothing commits or pushes on your behalf, so the working state isn't backed up until you run it yourself. Second, the automatic integrity pass runs on a clean exit; if a session crashes before it fires, the housekeeping is deferred until the next session's start-up backstop catches it — and it always runs against current state, never retroactively for the crashed session's specific changes.

The practical takeaway: closing the window loses no work and still tidies the vault; running /librarian session-close yourself is how you tidy on demand and get the backup offer. Backup, run periodically by hand, is the one habit the automation deliberately leaves to you.

How this ties back to memory and the binder

Close-out is where a session's contributions settle: episodic memory records what happened, the binder's generated pages and its situating card re-derive from the plans you touched, and indexes refresh — so the next session in this project starts already oriented. The session keeps your place while you work (checkpoints) and coordinates with peers (the registry); close is where it hands the durable residue back to context and memory.

Reference

  • Explicit — /librarian session-close: auto-detects scope (solo / scoped / reconciler), chains the cleanup chores, advisory semantics, always exits cleanly.
  • The receipt — a dated session-close-*.md log in ~/.local/state/brain-stem/logs/ (machine-local, not the vault).
