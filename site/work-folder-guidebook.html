System personalization · setting up your non-coding work

The work-folder guidebook

For any non-coding work — consulting, program management, research, writing — the work folder is where you work. This is the practical guide: how brain-stem supports a project directory from the moment you create it, the difference between a flat and a master layout, and two worked examples of what a personalized setup actually looks like.

A guidebook. What gets scaffolded, how to create a project directory, and a fully worked personalization example.

Coding work? You may not need this

A lot of people use Claude Code primarily for coding. If that's you, you may not need the work folder at all: launch Claude Code from your code repository and add a CLAUDE.md, and Claude is situated well enough to be productive. The work folder is for the other case — when you want to lean on Claude Code for non-coding work too. There it pays to build out a proper project directory in a Claude-native folder structure, which is exactly what the work folder gives you — together with the support skills and governance that keep that structure current.

One folder per project, launched from inside it

Each non-coding project lives in its own directory under ~/work/. You launch Claude Code from inside that directory — that's what tells the system which project you're in and loads its context. (Never launch from your home folder; always launch from the project.) The work folder is the top-level directory that holds one sub-folder per project.

For the conceptual framing of coding versus non-coding work surfaces and how either rolls up into the project binder, see Project organization. This page is the practical how-to for setting a non-coding project up.

Flat or master: match the layout to your work

A project directory comes in two layouts. The difference is simply whether the project has one stream of work or several.

Flat

One stream of work

A single project with one set of deliverables and reference material. The default, and the right choice for most things — a report, a campaign, a piece of writing.

Master + sub-directories

Several parallel streams

One umbrella project that contains several distinct workstreams, each with its own deliverables and reference. The right choice when you're juggling parallel efforts under one roof.

What the system scaffolds for you

When you register a project, brain-stem writes a small, deliberate starting structure — and nothing more. It scaffolds the shape; you own the content.

A flat project — five things

~/work/<project>/
├── CLAUDE.md      ← thin identity + an auto-maintained "what lives where" map + a pointer to the binder
├── README.md      ← scope, outcome, definition of done (yours to write)
├── updates.md     ← an append-only running log (seeded with one line; yours and Claude's to append)
├── deliverables/  ← polished, audience-facing work   (gets an auto-maintained _index.md)
└── reference/     ← raw notes and source material     (gets an auto-maintained _index.md)

The CLAUDE.md carries a small machine-maintained block — a "what lives where" directory map the system keeps current as the folder grows — plus a pointer to the project's binder. It does not import a cover page; in this layout there is no hub.md in your work folder at all.

A master project — a thin top, then one folder per stream

~/work/<project>/
├── CLAUDE.md          ← master identity + the work-map + binder pointer (no top-level deliverables/reference)
├── README.md
├── updates.md
└── <first-stream>/    ← a workstream — an organizational unit, never its own CLAUDE.md
    ├── README.md      ← carries a one-line "launch from the master" note
    ├── deliverables/  (auto _index.md)
    └── reference/     (auto _index.md)

The master top holds only identity files; the deliverables/ and reference/ folders live inside each stream. Sub-folders are organizational units — they never get their own CLAUDE.md or cover page; you launch from the master.

Where the cover page actually lives

The one cover page for the project — its hub.md — is not in your work folder. It's minted in the project's binder over in the plan tree (~/.claude-plans/_projects/<project>/hub.md) at registration. The _index.md files in deliverables/ and reference/ and the work-map block in CLAUDE.md are the first files the system ever generates inside ~/work/; everything else there is yours to write.

How to create one

You don't run any of these skills yourself, and you don't have to memorize a command. In any Claude Code session, just tell Claude what you want it to do — for example: "Use your governance skills to help me set up a new non-coding project in my work folder for a Q3 pricing study." Claude reaches for the right capability on its own (the /govern register family), proposes a structure for you to approve, and writes only once you say go. Your job is to describe the project; its job is to build and register the shape.

It runs in two stages. Simply making a folder and launching Claude inside it isn't enough — an unregistered work folder is deliberately left untouched until it's registered. So you first create & register the project, then launch from it and make it yours. The one hard rule: the folder sits exactly one level under ~/work.

Stage 1 · Create & register

What each step means, and who does it — you describe, Claude builds:

1 · You describe itIn a session, ask Claude to set up a new non-coding project in your work folder. Name the project, and say whether it's one stream of work or several. There's no command for you to type.
→
2 · Claude proposesIt picks the right registration and proposes the structure — flat, or master with a first stream — for you to redline field by field. Nothing is written to disk yet.
→
3 · It registersOn your go-ahead it scaffolds the folder, mints the project's binder hub and the _index.md files, and adds a vault-view routing rule — so the project is governed from its very first file.

Under the hood that's /govern register --kind project --layout flat (or --layout master --first-sub <name> for the master shape) — but you never have to type it. You ask; Claude runs it.

Stage 2 · Launch & personalize

To work in the project, launch Claude Code from inside its directory — cd ~/work/<project>, then start Claude from there. That launch directory is what tells the system which project you're in and loads its context, every session from now on. From here the folder is yours to grow, and brain-stem keeps it governed as you do — no extra ceremony required:

You add a folder

It gets an index

Drop in a new folder and brain-stem auto-creates and maintains its _index.md — a "what lives here" map kept current as the folder grows, so the directory never becomes an opaque pile.

You add a kind of document

Claude helps you govern it

Want every meeting note or status file to share one shape? Tell Claude, and it registers that file type and its contract — so each file of the kind is validated and governed natively, exactly like the types brain-stem ships.

You add a folder type

Register it once

Need folders the base scaffold doesn't include — a people/ directory, a SoW/ folder? Ask Claude to register the new folder type, and from then on it's part of your governed structure.

You wire up a feed

It fills itself

Register a vault-writer and a structure keeps itself current — transcripts, emails, or research landing in the right place on a schedule, safely de-duplicated. See the vault-writer system.

Growing or reshaping the project later works the same way: describe the change and Claude runs the right registration for you. To add another workstream to a master, or to promote a stream into its own top-level project after you move it, the registrations it reaches for are: 

# grow a master with another stream
/govern register --kind project --under <project> --add-sub <name>

# promote a stream to its own top-level project (after you move it)
/govern register --kind project --adopt

Worked example 1 — a flat project

Say you're writing an independent market study: one report, one stream of work. A flat layout is all you need. 

~/work/market-study/
├── CLAUDE.md      ← "this is the Q3 SaaS pricing study"; work-map; binder pointer
├── README.md      ← the question, the scope, what "done" looks like
├── updates.md     ← running log: what you learned each day
├── deliverables/
│   ├── exec-summary.md
│   └── full-report.md
└── reference/
    ├── interview-notes/
    ├── pricing-data.md
    └── competitor-scan.md

Launch Claude from ~/work/market-study/ and it already knows the question, the scope, and what you've gathered. Drafting the exec summary becomes a one-line prompt, because the report's reference material is right there.

Worked example 2 — a master project, personalized

Now meet Johnny, a project manager. He works across several workstreams that belong to different teams, and he uses Claude Code and brain-stem to track and document progress, stay on top of risks and blockers, and keep his teams moving toward delivery. His project is a master with one sub-folder per workstream — and he's extended the base scaffold with a couple of folder types of his own: 

~/work/program-delivery/
├── CLAUDE.md
├── README.md
├── updates.md
├── workstream-atlas/
│   ├── README.md          ← scaffolded; a one-line "launch from the master" note
│   ├── people/            ← one file per team member: who they are, role, responsibilities
│   ├── meeting-notes/     ← one file per meeting, in Johnny's preferred structure
│   ├── deliverables/
│   │   ├── project-brief.md   ← synopsis, key stakeholders, core goals
│   │   ├── prd.md             ← the requirements for this workstream
│   │   ├── daily-status.md    ← append-only ledger: each day's activities, blockers, risks, top action items
│   │   └── roadmap.md         ← stages and the key activities in each
│   └── reference/
│       └── ...source material for this workstream...
└── workstream-borealis/
    └── ...the same shape...

The people/ and meeting-notes/ folders aren't part of the default scaffold — Johnny registered them as folder types of his own (see Governance personalization), and he wrote a file-type contract so every file of a given kind lands in the same structure. The base scaffold gave him deliverables/ and reference/; he built the rest to fit his work.

How Johnny made it run itself

Everything in that directory is kept current by brain-stem capabilities, because Johnny composed a capture setup on top of the vault-writer system. Here's the shape of what he built — and, honestly, which parts are his to build versus the system's to provide:

  • Connectors he wired up pull his Granola meeting transcripts and his Gmail. (The connectors themselves are yours to build on the substrate — the system provides the safe-writing and de-duplication machinery they feed.)
  • Deterministic routing. For each workstream he handed Claude the list of people and their emails, so transcripts and messages get sorted to the right workstream by rule, not guesswork.
  • Better meeting notes than the raw transcript. He registered a meeting-notes file-type contract and attached a prompt that points Claude at that workstream's project brief and PRD as reference, so each transcript is processed into a richer, more accurate note than Granola gave him.
  • One structured spine. Meeting notes land as their own reference assets; meeting notes and emails together feed a SQLite database — workstream, progress updates, new action items and their due dates, and the person who owns each line — where the records are de-duplicated and cleaned before they go anywhere.
  • One writer, many files. From that clean spine, a single agentic writer fans out across all the files Johnny organizes his work in — appending to the daily status ledger, updating the roadmap, amending the brief as requirements evolve — and every change it applies is recorded in an append-only, immutable audit trail. His updates.md carries the running, human-readable log of where each workstream stands.

Johnny set up meeting transcript and email ingestion to run on an hourly cadence throughout each work day. As each email and transcript arrives, raw content is processed into his structures automatically. (Scheduling a capture run like this is something you assemble — see Orchestration & scheduling.)

Why this is the whole point

Because Johnny launches Claude Code from this work directory, it has immediate access to the most up-to-date context and reference material for every workstream he owns. So when he needs a new deliverable — a presentation, a PM dashboard, a strategic analysis — he produces it with simple prompting, quickly and accurately, because the context is already loaded. Without this setup, he'd have to re-share all of that material himself in every single session he tasked Claude with work — paying in tokens, time, and unproductive manual effort, over and over. The work folder is what turns that recurring tax into a one-time setup.
