The memory model

Out of the box, Claude Code starts every session blank, and the one file it auto-loads is silently capped. brain-stem's memory is the answer: a small, curated, classified set of on-disk notes the assistant reads back at the start of every session — deliberately not a catch-everything pile that rots.

Project memory lives at ~/.claude/projects/<project>/memory/; global memory at ~/.claude/CLAUDE.md + ~/.claude/rules/.

Three kinds of memory

Memory is split into three kinds rather than one undifferentiated pile. The kind is declared by the author at the moment of writing, never guessed afterward.

KindHoldsDecays?
SemanticTimeless facts, preferences, identity, naming conventions — the standing knowledge of who you are and how your world is named.Re-validated periodically
ProceduralHow-to knowledge: workflow rules plus the reasoning behind them, including lessons learned from incidents.Re-validated periodically
EpisodicDated records of what happened in a specific work session; time-anchored.Never decays

The split lands where independent memory science already landed (the episodic/semantic and declarative/procedural distinctions, carried onto language agents), which is reassurance that the shape is convergent rather than invented. A fourth "reflective" type was deliberately dropped: reflection is a process, and its output already belongs in Procedural (it rides the feedback_ provenance prefix).

Operational vs contextual

Memory straddles brain-stem's central axis. Semantic + Procedural are operational — they govern how the assistant behaves. Episodic is contextual — it records what happened. The project binder then continues that contextual job outside the formal memory structure. Part of that binder now loads on its own: a generated project-context card (the situating card, _situating.md) is auto-loaded at session start alongside memory and placed before the assistant's first turn — a machine-derived summary of the project's plans, their combined status, the active focus, and the latest handoff. See the project-binder page for how it is generated and what it carries.

How it loads — the hot index and the cap

The native constraint is exact and unforgiving: Claude Code auto-loads a memory file but reads only the first 200 lines or the first 25,600 bytes (~25 KB), whichever comes first — and anything past either limit is silently dropped, with no error and no truncation marker. There is no include mechanism to reach past it.

So MEMORY.md is built as a hot index — a read-replica table of contents. Each line is a one-line pointer; the full detail sits in per-topic files read on demand. The cap bites on the small index, not the detail, which is exactly why the index has to stay small. The byte cap governs — a dense, fact-first index hits 25 KB before it hits 200 lines.

Keeping the index loadable

Two pieces of machinery watch the same cap (both read it from one config slot, so they never drift):

  • A write-time guard (advisory) — just before a memory-index write lands, it checks the caps and, if a write would breach one, injects a warning naming the breached limit — then lets the write proceed. It never blocks.
  • A budget monitor (record-only) — after a session, it grades the index GREEN (<75%) / YELLOW (75–89%) / RED (≥90% of the byte cap), counting lines both raw and with comments stripped and gating on whichever is larger, so hiding bulk in comments buys no headroom. It records; it prunes nothing.
Documented, not yet built

The planned overflow path — a cold-spill that relocates the coldest non-episodic entries (Procedural first, then Semantic, never Episodic) to an on-demand MEMORY-archive.md, leaving a pointer in the index — is designed but not yet implemented. It is volume-gated (built when a real index nears the cap), not calendar-based, and the config records this honestly. Today, the guard warns and the monitor grades; the spill itself is future work.

How memory is written and classified

Every memory note carries two labels, fixed at write time:

LabelSaysValues
Provenance prefix (filename)where the fact came fromuser_ · feedback_ · project_ · reference_ · episode_
Retrieval type: (header)which kind it issemantic · procedural · episodic

The prefix implies a default kind (user_/reference_ → semantic; feedback_ → procedural; episode_ → episodic), but the declared type: is the source of truth. The contract for these fields is a shipped schema (schemas/memory-schema.json), enforced by the advisory write-time guard — the assistant that learned the fact states its kind directly; no hidden classifier guesses after the fact. The librarian only validates (flags a note with no index pointer, a stale note, an over-budget index); it never re-sorts sections or re-derives a note's type.

Episodic memory is a single chronicle

Episodic memory is the one fully-automatic tier: at session close, a hook appends one harvested row — anchor, files touched, next-session pointer, a short summary — to a single append-only episodic-chronicle.md, with no model call. It does not create a file per session. The ## Episodic section of MEMORY.md is a single pointer line to that chronicle.

Curation and promotion

  • Staleness — non-episodic notes carry a last_validated date and are flagged for re-validation on a 180-day interval (expired at 360); episodic never decays. All propose-only — nothing is deleted automatically.
  • Globalize — a project memory marked scope: global can be promoted up into ~/.claude/rules/, transformed into a schema-validated rule. It is propose-by-default (an explicit confirm applies it), idempotent, and writes only to rules/ — never to CLAUDE.md, which stays hand-edited.

Why this design

ChoiceThe reasoning
A small hot index + cold topic filesThe native ~25 KB/200-line cap with no include mechanism forces this shape — it's optimal by constraint.
Three declared kindsClassification at write-time resists the noise that sinks catch-everything memory; it converges with established memory science.
Classification declared, never inferredThe author knows the kind; a separate after-the-fact classifier would silently misfile.
Globalize writes only to rules/CLAUDE.md is yours to hand-edit; no automated process may write it.

Reference

  • ~/.claude/projects/<project>/memory/MEMORY.md — the auto-loaded hot index (fixed Semantic → Procedural → Episodic order) + per-topic detail files.
  • schemas/memory-schema.json — the shipped frontmatter contract (the three-kind enum, required last_validated).
  • ~/.claude/CLAUDE.md + ~/.claude/rules/ — the two global surfaces; rules/ is the globalize destination.
  • The byte-first write-time guard and the post-session budget monitor — both read the cap thresholds from one config slot.
  • episodic-chronicle.md — the single append-only episodic record, fed one row per session by a SessionEnd hook.
