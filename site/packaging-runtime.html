What "the foundation" is, and why it has to be installed

Out of the box, Claude Code's assistant is a blank slate — no rules, no automations, no reusable procedures. "The foundation" is the curated bundle that turns that blank slate into a disciplined workspace. It contains four kinds of things:

IngredientPlain meaning
SettingsA single configuration file that tells Claude Code how to behave.
HooksSmall scripts the program runs automatically at moments in a session (for example, when one starts).
SkillsReusable, named procedures you invoke on demand (/onboard, /govern register, /librarian).
RulesMachine-readable rule files that encode the foundation's conventions.

"Installing" means copying that bundle out of the downloaded source folder into the one directory Claude Code reads from when it starts up.

~/.claude — where Claude lives (and CLAUDE_HOME, the knob that picks it)

When Claude Code launches, it looks in one folder for everything it should load: its settings, automation scripts, skills, and rules. By long-standing convention that folder is ~/.claude in your home directory. One environment variable, CLAUDE_HOME, changes the destination — useful for testing an install in a throwaway location. If it is unset, the documented default is ~/.claude. The mental model: ~/.claude is "where Claude lives," and CLAUDE_HOME is the override that says "put it somewhere else."

Dry-run by default, then --apply

The riskiest moment in any install is the first time it writes files. So the plain command — bash install.sh with no extra words — does not install anything. It prints an action plan (structured JSON naming every folder it would create, every set of files it would copy, and a reason for each), then exits having written zero files. The plan is emitted only after every safety check passes but before the first folder is created — so the preview reflects fully-validated state yet cannot accidentally mutate anything. Pipe it through jq . to read it comfortably.

To actually perform the copy, re-run the same command with one word added — --apply, the explicit opt-in that flips preview mode into do-it-for-real mode. There is a deliberate safety asymmetry:

ModeIf CLAUDE_HOME is not setWhy
Dry run (bash install.sh)Quietly assumes ~/.claude to render a plausible preview, and records a claude_home_defaulted flag disclosing that it guessed.A preview writes nothing, so a sensible default is harmless.
Apply (bash install.sh --apply)Refuses — it stops with an error and exits without writing.Real files must never land in a folder you did not consciously choose.

The principle: a preview may assume a target; a real write may not. Files only ever land where you deliberately pointed them.

Pre-flight gates: the checks that run before any file is copied

Before the installer creates anything, it runs a sequence of quick safety checks, internally called gates. Each answers one yes/no question and refuses to proceed if the answer is dangerous. The installer is paranoid on purpose — most gates exist to stop you from clobbering something you care about.

The gate's questionIf the answer is dangerous
Are you running as the all-powerful root account?Refused — too much blast radius for a personal install.
Does the target folder hold someone else's unrelated Claude setup?Refused, unless you explicitly force it with a confirmation flag.
Have the foundation's own files in an existing target drifted from the shipped version?It depends why — see two reasons a file can differ. An older shipped version is replaced cleanly; genuinely edited content is preserved and a real apply demands a typed override.
Does the target path secretly tunnel into your synced notes vault via a symlink?Refused outright — that path is protected, with no override.
Does the plan folder already contain an existing plan tree?Refused unless you pass an acknowledgement flag, so it never silently overlays existing plan files.
Would merging the new settings silently delete keys from your existing configuration?Refused — your existing settings are never quietly discarded.
If a destructive step is pending, can it first prove a backup will work — and take one?Refused — it demands a writable backup location, verifies it round-trips, and copies your existing settings into a timestamped settings.json.pre-install-<timestamp> first. No provable backup, no install.
Do you have the helper programs the install needs (e.g. jq)?Refused if any are missing.

Upgrading in place — two reasons a file can differ

The drift gate raises a question: if the installer refuses when a file's fingerprint does not match the receipt, how does it ever upgrade an existing install — where, by definition, your files differ from the new ones? The answer is that "differs from the shipped version" has two completely different causes, and the installer must tell them apart.

StateWhat it meansWhat the installer does
Version-delta — the on-disk bytes match a known previous releaseSimply an older version of a foundation file, sitting where the upgrade is supposed to replace it. Not something you changed.Take the new version cleanly. A normal upgrade, no confirmation needed.
Foreign / edited — the on-disk bytes match no known releaseYou (or some tool) changed this file. The foundation has no record of these exact bytes, so it assumes they are yours.Preserve your version. The new version is laid down, but your copy is saved alongside as <file>.foundation-local. A real apply will not silently overwrite it without a typed confirmation.

What makes the distinction possible is the per-release baseline archive — a stored fingerprint list for every version the foundation ever shipped (under governance/baselines/). To classify a differing file, the installer checks its fingerprint against every archived release: a match means "old shipped version" (replace); no match means "your edit" (preserve). A legacy install (made before the version stamp) is brought current by walking the shipped file list one file at a time, so every managed file is examined — not just some subset of directories.

The delivery gate — an upgrade won't claim success if it fell short

Before writing the completion stamp that records an upgrade as finished, the installer verifies that every managed file it shipped actually reached the new version. The check is disposition-aware: it does not demand that files you edited (deliberately preserved) or merged (like your overlay) match the shipped fingerprint — only that the files meant to take the new version did. If any file that should have updated is still stale, it fails closed: stops with exit code 56 ("delivery shortfall"), writes no completion stamp, advances no baseline. Re-running bash install.sh --apply picks up the unfinished work and converges. A half-delivered home is never recorded as a finished upgrade.

The manifest (ship-list): the receipt of exactly what was installed

When you install, a record is kept of every single file laid down — its location, size, permission setting, and a fingerprint (a SHA-256 content hash: a short string derived from a file's exact contents, so that if even one character changes, the fingerprint changes too — a tamper-evident seal). This record is the manifest, installed at ~/.claude/governance/foundation-manifest.json. Each entry records path, sha256, mode, and size. The manifest does three jobs:

  1. It is the authoritative answer to "what does a correct install look like?"
  2. It lets a later check detect tampering — an edited shipped file no longer matches its fingerprint.
  3. It tells the uninstaller precisely which files are safe to remove.
Why the manifest is generated, not hand-written

A hand-maintained file list always drifts — someone adds a file and forgets to list it, or removes one and leaves a stale entry. So a generator walks the actual source tree and emits the manifest automatically. Because the same walk defines both what the installer ships and what the manifest records, the two cannot silently disagree; a release-time check fails the build on any mismatch. The receipt is a photograph of the truth, taken automatically — not a list someone tried to keep current by hand.

Uninstall: reversing the install without destroying your work

Over time your files end up living next to the foundation's files in the same folders, so a blunt "delete everything in ~/.claude" would take your edits with it. The uninstaller (bash uninstall.sh) reads the manifest and goes file by file:

Comparison resultWhat it meansWhat the uninstaller does
Fingerprint matchesUntouched, original foundation content.Safely removes it.
Fingerprint differsYou edited this file.Preserves it and reports it — never silently deleted.
File it never createdYour own content.Left alone entirely.

Your personal edits survive an uninstall by default. The safety net is a mandatory backup taken first: before removing anything, it copies the entire target folder into a timestamped .pre-uninstall-<timestamp>/. There is no preview / dry-run mode for uninstall (the --dry-run flag is listed as deferred in the script header) — the backup is the safety mechanism instead.

Part of what the foundation can set up is scheduled background jobs on macOS, managed by the system's service manager launchd and each identified by a label. The foundation reserves a private name-space: every one of its jobs is named starting with com.brain-stem. Uninstall stops and removes only jobs in that reserved name-space — and actively refuses to touch any job whose name merely looks similar but sits outside the prefix, aborting rather than risk stopping something unrelated. This is why uninstall fully reverses the install: it pulls down the timers, not just the files.

Where runtime state lives — and why it is kept out of ~/.claude

There is an important difference between the foundation's program (the scripts, skills, settings, and rules — what you installed, in ~/.claude) and its runtime state (the working data those scripts produce: session bookmarks, processing records, scratch files, coordination locks). The foundation deliberately stores runtime state in standard locations outside ~/.claude, following the XDG Base Directory convention — so uninstalling or reinstalling the program never endangers your accumulated working data. The state is split into two roots by how much you would care if it vanished:

RootLocationHolds
Durable~/.local/share/brain-stem/The write-activity database (manifest.sqlite) and per-day processing records — expensive or impossible to recreate; lands where backup software looks.
Ephemeral~/.local/state/brain-stem/Throwaway staging scratch, per-session checkpoint files under sessions/<session-id>/, and the .coordination/ lock files that keep multiple Claude windows from colliding — cheap to throw away and rebuild.

The mental image: "the app" and "the app's notebook" are kept in different drawers on purpose. (Plan files live in a third place again — ~/.claude-plans/ — which is neither the program folder nor a runtime-state root.)

Two more records round out the design. Every real install writes a dated provenance log into ~/.claude/logs/ recording the facts of that install, including the folder it targeted. It is durable evidence of what was done, and it is how the uninstaller double-checks it is pointed at a genuine install — if there is no install log, the uninstaller refuses. And Claude Code reads a single config file, ~/.claude/settings.json; the installer deep-merges the foundation's settings into it field by field rather than overwriting — atomically (it fully succeeds or leaves the original untouched), and refusing to proceed if the merge would silently delete any of your existing keys.

The failure modes this packaging design defends against

Every safety property exists because a simpler design fails in a concrete, painful way. Stated plainly:

Design choiceThe failure it prevents
Dry-run by defaultAn installer that writes immediately gives you no chance to inspect what lands where. The preview exits before the first folder is created — provably write-free.
Apply refuses an unset targetReal files landing in a folder you did not consciously choose.
Drift detection on an existing installA reinstall blindly overwriting foundation files you had genuinely edited; the fingerprint check distinguishes an old shipped version from a real edit.
Disposition-aware delivery gateAn upgrade recording itself as finished while some files silently stayed out of date (exit code 56 stops it).
Backup-proof-of-life before any destructive stepA destructive overwrite running with no working backup.
Generated (not hand-edited) manifestA hand-kept file list drifting; a release-gate parity check fails the build on any mismatch.
Fingerprint walk on uninstallBlunt deletion taking your own edits with it.
Mandatory backup before uninstallAn irreversible mistake — the whole target is copied first, so even a surprise is recoverable.
com.brain-stem.* label fenceA careless cleanup stopping unrelated background jobs.
Two-root state splitProgram churn (uninstall/reinstall) endangering your accumulated working data.
Provenance-log checkAn uninstaller pointed at the wrong or never-installed folder.
settings.json deep-mergeOverwriting your existing configuration.
None of this is novel — that's the point

These are the same patterns mature package and dotfile managers settled on independently: a generated receipt of installed files (Homebrew, Nix, apt/dpkg), preserving files the user changed rather than clobbering them (chezmoi), and keeping working state out of the program folder sorted into durable and ephemeral roots (the XDG specification). Where brain-stem goes one step further than the typical installer is that it emits a provably write-free action plan before the first write — you see the entire set of file operations, having changed nothing, before you authorize one.
