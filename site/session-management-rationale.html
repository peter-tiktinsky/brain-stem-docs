Sessions

Sessions

A session is one continuous conversation, and its working memory is perishable: when a long session fills the assistant's fixed window, the platform compacts it — replaces the verbatim history with a summary and discards the originals CC context-window. The Sessions pillar wraps a session so this loss never costs your place. It is the most constraint-forced pillar in the system — the constraints did most of the designing — with only two narrow convergence echoes.

brain-stem choseoverbecauseground
Save the checkpoint before compaction, restore it at the next start (two hooks) Handing the saved state back at the compaction moment itself The platform allows context injection at session start and message-submit, but not at the pre-compaction moment CC hooks — so the restore necessarily lands at the next start. This is also the classic checkpoint-then-resume shape ARIES LangGraph. constraint convergence
Coordinate concurrent sessions through a machine-local registry Keeping the coordination file in the synced vault "for cross-machine safety" That guarantee is a phantom: liveness checks and the write lock are physically local to one machine and mean nothing on another Kleppmann; a hot synced file also invites conflict-copies. So coordination stays local, written atomically, serialized with a lock lockf. constraint
No checkpoint schema and no parser — re-inject the block as plain text A strict schema plus a parser for the checkpoint The only consumer re-injects the block as text, with no parser in the loop. A formal schema would create a second source of truth that could drift, for zero practical benefit; a light count-the-structure heuristic is the deliberate substitute. design judgment
A context-pressure ladder that escalates to a hard stop-gate Relying on the person to remember to save A long session fills its window silently — there is no felt warning at the moment detail is about to be trimmed. The ladder (silent autosave → nudge → mandate → stop-gate) forces a rich save before the detail is lost CC hooks. constraint
Fix the stop-gate's band boundaries in code, not settings Making the hard boundary user-tunable Making the hard safety boundary tunable would let an adopter weaken the very gate that prevents losing state. Only the two gentle nudge thresholds are settings-driven. constraint
Three checkpoint writers, one freshest-richest-wins precedence on one shared per-session file Multiple writers writing multiple files, or no precedence at all Targeting one shared per-session file makes the precedence a real contest rather than three files in three places — and guarantees two concurrent conversations can never overwrite each other's place. The pre-compaction fallback is no-AI-cost because that hook runs under a tight timeout. constraint
Session-close as an advisory /librarian capability that always exits clean A blocking close, or every session running the global shared sweep A blocking close can wedge a session; deferring the shared sweep to whoever closes last stops two closers trampling each other or double-running cleanup. Scope is auto-detected (solo / scoped / reconciler); a failed chore is logged and the flow continues. constraint
Write the close receipt unconditionally, relocated out of the indexed vault A receipt in vault/Logs/, or a conditional emit surfaced by a banner A vault receipt leaks into search and the graph; a conditional emit surfaced by a banner is unimplementable on the shipped surface and would collide with the one-file-per-session contract. So the receipt is written unconditionally, under the XDG state home XDG. convergence
Capture the end-of-session episode as a thin, no-LLM row A CoALA-STAR narrative body CoALA A per-session STAR narrative would be four static sentences with almost nothing interpolated — boilerplate with no retrieval value. Capture is harvested, not authored — the row-now / enrich-later split Zep LangMem endorse — and the 5-second no-LLM hook can't tokenize, so the rich summary is deferred to session-close. constraint convergence

Decision quality

The decision-quality protocol

The decision-quality protocol addresses a failure mode distinct from the session-continuity problems above. When Claude Code presents options — scope, approach, design, sequencing — the default behavior is to rationalize the first option set the model surfaces rather than expand the space. This is not a context-window problem; it is a deliberate-reasoning problem. The protocol adds a structured research pass before options are published to the user.

What shipped

Phase 1 shipped as a PreToolUse hook on AskUserQuestion (the pre-asq-guard). When the hook detects a substantive option set — the substance heuristic keys on whether the message presents numbered alternatives or a fork-in-the-road framing — it injects the 4-element protocol into the turn as additionalContext, nudging the model to run the research pass before publishing the options. It fires as an advisory: the options are never blocked, only preceded by a nudge to run the pass first.

The 4-element pass the nudge asks for:

1

Re-read the project goals

Read the active plan's spec, manifest, and recent handoff before surfacing options. The right answer is often constrained by criteria already documented.

2

Check inter-project dependencies

Scan adjacent plans for conflicts, synergies, or shared infrastructure that changes the decision shape. An option that looks clean in isolation may block a parallel plan.

3

Read the live vault state

Read current files; do not trust memory. Memories are point-in-time observations; the code on disk is authoritative. Many "debates" dissolve when someone reads the file.

4

Generate one unconsidered option

Produce one option not already on the table. This is the load-bearing step — it expands the option space rather than rationalizing the existing list. Without it, the protocol collapses into performative deliberation.

Why this shape

brain-stem choseoverbecauseground
A substance heuristic on AskUserQuestion that fires before options are published A standing instruction in CLAUDE.md to "always research options first" Standing prose instructions degrade over a handful of sessions and get rationalized away under pressure. A hook fires regardless of whether the model is cooperating — the same constraint that drives the governance engine's write-time interception. constraint
Inject as additionalContext — advisory, never blocking A hard block that refuses to surface options until the pass is complete The goal is a better option set, not a gate that trains users to work around it. The nudge is enough: empirically, a "research each option thoroughly" prompt overrides the model's first-presented options in nearly every session — the advisory produces the behavior without the friction of a block. design judgment
The +1 unconsidered-option step as the mandatory fourth element A simpler "re-read goals + read files" two-step Re-reading existing material rationalizes; generating one new option expands. Tree-of-Thought and Reflexion research demonstrate that deliberate divergence from the initial hypothesis is the mechanism that produces better answers — not reviewing the same set more carefully. The step is structural, not stylistic. convergence
Skip-conditions that exclude trivial prompts (yes/no, color choice, clarifications) Firing the protocol on every AskUserQuestion call Alert fatigue from a protocol that fires on "do you want to use tabs or spaces?" would train users to dismiss every nudge. The skip-conditions are a deliberate scope gate — the same data-gated principle that governs how governance rules earn the right to get strict. design judgment
Phase 1 advisory shipped; Phase 2 blocking opt-in gated to post-release telemetry Shipping the blocking version first, or deferring the entire protocol An advisory that ships is more valuable than a block that waits. Phase 2 (hook promotes from nudge to block for substantive option sets) is deferred until Phase 1 telemetry confirms the nudge fires on the right cases and the false-positive rate is low enough to block on. design judgment
