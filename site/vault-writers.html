The vault-writer system

The vault-writer system is the machinery that turns outside information into structured, reusable assets in your vault — and keeps doing it on its own. It ships the hard parts (safe writing, deduplication, an audit trail, a deterministic-and-optional-AI processing path); you supply the connectors that decide what to pull and how to shape it.

A deep page. What the system guarantees, the design principles for building on it, and where the lines between "shipped" and "you build it" actually fall.

The pipeline, end to end

Whatever the source — an email, a meeting transcript, a scrape — the path into your vault is always the same shape. A connector does the fetching and shaping, then hands the result to a pipeline that does the writing. The connector never writes a destination file itself; it emits a packet, and a single program applies it.

SourceEmail, transcript, scrape — pulled by a connector you register.
→
Staging packetA content-addressed JSON packet, named by the SHA-256 of its body. Holds the body + where it should land + an output type.
→
Writer-reconcilerThe sole writer. De-duplicates by content hash, picks the winner, and writes atomically — under an OS lock so two runs never collide.
→
Destination(s)A note, a JSON file, or a SQLite table — plus an append-only manifest row recording the write.

Two properties of that middle step do a lot of work. First, the writer-reconciler is the only thing that writes a destination — enforced with an exclusive lock, not just by convention — so you never get two processes racing on the same file. Second, your hand-edits always win: if you've edited a file the system also maintains, a two-signal check (a frontmatter marker plus a content-hash drift) detects it and the reconciler preserves your edit instead of clobbering it.

Where everything lives (and what's permanent)

Raw source is retained under ~/.local/share/brain-stem/vault-writers/raw/; a per-day processing ledger and the durable manifest live alongside it (manifest.sqlite, a WAL-mode writes table, append-only). The destinations are your actual files. The only ephemeral piece is the in-flight staging packet at ~/.local/state/brain-stem/vault-staging/<writer-id>/<sha256>.json, which is deleted once it's applied. So your raw inputs, your processed records, the audit log, and your outputs are all kept permanently and locally — only the transient hand-off is thrown away.

It doesn't only write notes

Each packet carries an output type, and that's what lets one capture land in more than one shape. The same processed information can be a readable note and a row in a queryable table.

Output typeLands asGood for
markdownA note in your vault.Anything you read — meeting notes, summaries, briefs.
jsonA structured JSON file.Records you'll process again, or feed to other tooling.
sqlite / dbA SQLite (or external) database file.Anything you'll query later or wire to a dashboard.

This is the quiet superpower: if you ever want to query your captured information, or use it as the data behind a web page, a dashboard, or any product that needs structured input, the structured copy is already there — you don't have to go re-extract it from prose.

Build it deterministic-first

The most important design principle for a connector is this: when you're turning unstructured content into a formal structure, make the processing as deterministic as you can. The system's own posture models this — the default path is a plain mechanical write with no model in the loop, and the AI lane is something you opt into. There are four reasons it pays off:

1

Correct processing

Deterministic rules behave the same way every time — essential when the same fact can arrive from more than one source.

2

Trustworthy formatting

If you want information to land in a consistent, predictable shape, a rule gives you that; a model's phrasing drifts.

3

Token economics

Repetitive work shouldn't lean on a heavy model. The more structure you give the ingestion, the less token tax you pay.

4

It scales quietly

Deterministic steps run on a schedule without supervision and without surprises — the right base for always-on capture.

Token cost: the optimization lever is in your connector, not the writer

The token-economy argument is real, but the lever is in your connector, not a setting on the writer. A connector is just a script: it can parse deterministically, or call a cheaper model (Sonnet rather than Opus), or even a local model at near-zero token cost, before it emits its packet. The pipeline's own optional AI step always runs with the ambient default model — it has no per-writer model dial today. So "use a cheap or local model for the repetitive part" is a way you design the connector, not a flag you toggle on the writer. The architecture makes it possible; you make it happen.

When to bring AI into the loop

Not everything is deterministic, and reasoning genuinely matters at certain moments. So the system keeps an opt-in LLM-reasoning lane: you can register an operator-authored prompt (a doc-amender-prompt) that runs through claude -p to merge an incoming packet into a destination intelligently. Even then it stays inside the safety model — the prompt composes a packet, it never writes the final file; the reconciler still does that. Think of it as: deterministic by default, model in the loop only where you decide it earns its keep.

Deterministic is enough

Action items by workstream

Pull action items from emails and transcripts and route each to the right workstream by handing Claude a list of the people on each one. A rule, not a judgment call.

Reach for the model

What resonated in the room

After a demo, gauge audience sentiment, or which of the ideas you floated landed best. That's a reading of meaning — the place to attach an LLM prompt.

You're never taxed for changing your mind

Here is the payoff that makes the whole thing worth setting up. Because the system keeps your raw content, holds onto the processed content, and stores the outputs — all permanently, all local — you are free to evolve what you want from it without re-architecting anything.

TodayPull action items from email. Trivial to start.
→
Add a sourceAlso pull them from meeting transcripts — no duplication to worry about, because you fan both into the same de-duplicating destination.
→
TomorrowWant a dashboard instead of a sheet? Reuse the SQLite the system already built for you.

You end up with all the Lego bricks you need to build pretty much anything you want — and, unlike a traditional system, you aren't penalised for deciding you want something different next week.

One honest note on de-duplication

"No duplicate action items" isn't magic — it's a setup choice. The system de-duplicates by content hash per source automatically; to collapse the same item arriving from two different connectors, you route both into one destination keyed on the item's identity, and a deterministic keyed upsert (replace-on-key) handles the rest. The pipeline gives you the de-dup machinery and the cross-source index; you point the sources at a shared keyed landing place to use it.

What ships, and what you build

It's worth being clear about the line, because it's where most of the leverage is. brain-stem ships the substrate; you compose the connectors on top of it.

Shipped (the substrate)You build (the connectors)
Writer registration (/govern register --kind writer); staging packets; the single writer-reconciler; the SQLite manifest; raw retention; fan-in and fan-out; the deterministic composer and the opt-in LLM lane. The fetchers themselves (e.g. a Gmail or Granola connector — the source labels are registration names, not shipped code); any source-to-workstream routing logic; a custom file-type contract for the shape you want; and any model calls your connector makes.

You register a writer, declare where its packets should land and in what shape, and the substrate takes it from there. A single agentic writer can fan out to many destination files at once — appending where things should append, updating where they should update — and every applied change is recorded in the append-only manifest, which is your immutable audit trail of what the system did.

Run it on a schedule

Capture is most useful when it's always on. The vault-writer pipeline is exactly the kind of work you'll want to run on a recurring basis — see Orchestration & scheduling for how scheduling works (and what's turnkey versus what you assemble yourself).
