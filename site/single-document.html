System governance · the single-document pillars

The single document

The first family of governance rules governs one file on its own — how it's identified and labeled (frontmatter, tags, name) and how its body is structured (the type contract). This page takes each of the four pillars in turn: the gap it fills in native Claude Code, the reasoning behind its shape, and exactly how it operates at the moment of a write.

The narrative companion to the foundation's frontmatter, tagging, naming, and file-type-contract rules. Every rule here is checked at write time against one merged contract.

One contract, composed from two files — read on every write

Each rule below is read the same way. At every write the engine merges foundation-master.json (the defaults every adopter receives) with overlay-master.json (your extensions, empty on a fresh install) and applies the union as the live contract. Foundation provides the baseline; your overlay extends the type registry, adds tag dimensions, declares new folders, and shadows a shipped default only where you've registered a stated _override_reason. The shape of that merge — sealed base, transparent overlay, overlay-wins with a stated reason — is itself the pattern Kustomize, git config, and the CSS cascade arrived at independently. See the governance engine.

Frontmatter — what a file declares

The gap in native Claude Code

Claude Code auto-loads a CLAUDE.md and reads any file you point it at — but to the platform every file is undifferentiated text. Nothing declares what a file is, what fields it must carry, or where it belongs; there is no write-time schema and no required-field check. So the structured handles that routing, auditing, and capture pipelines depend on simply don't exist until something supplies them. Frontmatter is that something.

Frontmatter is the YAML block at the top of every vault file, and the system reads it on every write to decide whether the file is well-formed and where it belongs. Fields like type, tags, and updated are not decoration — they are the structured surface the system reads (tags, covered next, are the surface you query). The payoff is that a file is legible to the machine the moment it lands, instead of being an opaque blob a later pass has to re-interpret.

The three compliance tiers

Every file type carries a tier — assigned per-type, not per-file — that decides what happens when a write fails its contract. This is the load-bearing idea: enforcement strength is matched to who authored the file.

TierOn failureDefault class of files
StrictWrite is denied; the missing fields are named.System-emitted files — scaffold output, captured content, scraper output, anything written by a skill without a human in the loop.
StandardWrite proceeds; an advisory surfaces at session close.User-authored content — your own notes, references, planning docs.
MinimalWrite proceeds without validation.Explicit opt-out — legacy imports, paste-buffer scratch, archived content outside the active lifecycle.

Foundation types are all Strict because they're machine-emitted; the user-authored types you register typically run at Standard. To drop a single file to Minimal, add tier: minimal to its frontmatter. The tiering is why the system can be uncompromising about generated files — which a human never sees and which downstream tools depend on — while staying gentle about the note you're actively writing. Blocking a person's save over a missing field would be hostile and would only train you to switch the guard off.

The foundation type registry

Foundation ships a type only when it has a foundation consumer — a hook, a skill, a mandate, or a contract. Everything else is left to your overlay so you shape your own vocabulary. The content types share a baseline of type, tags, updated; the operational log and the plan-tree ideation-brief carry their own minimal sets.

The registry is deliberately small — only types with a consumer. To validate a recurring document the foundation doesn't ship a type for, run /govern register --kind file-type; it writes the new type into your overlay under frontmatter.types with its required fields. To raise or shadow a foundation default, register the same type with an _override_reason — the system denies an override that lacks a stated reason, so every divergence carries its rationale. (Plan-tree files like spec.md and manifest.json are governed by the plans pillar, not this registry; see Plans.)

The folder-lineage convention

Folders don't carry frontmatter, so any lineage that lives at folder level — which cluster is this file part of — has to propagate down into the file. The convention: a file inside a structural cluster folder carries both the matching frontmatter field and the matching tag. Foundation ships the shape of these rules; your overlay's path_routing rules (declared at onboarding) define the cluster grammar — the path glob, the required lineage fields, the required tags, the tier, and any exempt sub-paths. A file under a cluster path without its lineage fields surfaces a placement advisory at write-time and at audit-time.

How it operates — the write-time sequence

When you write a vault file, the frontmatter pillar runs these checks in order:

  1. Type allowlist. The type: must be in the merged registry. On a Strict-tier write an unknown type is denied, the error naming the allowed types and pointing to /govern register; a Standard-tier write proceeds with a warning.
  2. Required fields. The universal three plus the per-type list must all be present. Missing fields on a Strict type are denied with the list of what to add.
  3. Folder placement. If the type declares an expected folder, a mismatch surfaces an advisory (never a block) — cross-folder reference docs and customized structures legitimately diverge.
  4. Folder-lineage fields. If the path matches an overlay lineage rule, the required fields and tags must be present — denied at Strict, advised at Standard.

The librarian's coverage audits then walk the vault periodically and surface recurring drift — types whose required fields you keep declining, folders accumulating placement advisories, two files both claiming to provide one capability. Those surface at session close; you triage.

Tagging — the query surface

The gap in native Claude Code

Claude Code ships no tagging system at all, and Obsidian's native #tags are freeform — no grammar, no controlled vocabulary, no cap. Left ungoverned, tags fragment into near-duplicate variants within months (#urgent, #high-priority, #priorityHigh) and the graph that was supposed to help you navigate becomes noise. Nothing keeps the vocabulary faceted or bounded. The tagging pillar supplies the grammar and the ceiling.

Tags are the surface you query — they drive the graph view, the filter pane, and the maps-of-content you navigate by. They're decoupled from frontmatter on purpose: a file is legible to the system through its fields even with no tags, and legible to you through its tags regardless of the rest.

The grammar, and two classes of dimension

Every tag matches #dimension/value — a registered dimension prefix, then a kebab-case value (the full shape is ^#[a-z][a-z0-9-]*/[a-z0-9][a-z0-9-]*$). A tag that fails the regex, or whose dimension isn't in the allowlist, is denied at write time, the denial enumerating the offenders so you can pick from the vocabulary, omit the tag, or register a new dimension. The flat freeform tag is rejected by construction — the dimension prefix is what keeps related values clustered and queryable.

ClassWhat it isWho stewards it · foundation ships
System-utilityMachine-emitted dimensions whose values are operational, not subjective.Skills, hooks, crons. Foundation ships #status/* and #log/* — cap-exempt.
User-facingDimensions you populate as you author and curate.You. Foundation ships only the shape and the cap; your overlay seeds the concrete axes at onboarding.

The split exists because the two classes drift differently. System-utility values come from skills with stable vocabularies, so fragmenting them into near-match variants would split the operational graph; they're governed by a log-subtype registry with two protections — a near-match denial (a value within Levenshtein distance 2 of an existing canonical is refused with a "did you mean…?") and explicit first-emission registration when a skill first emits a subtype. User-facing values come from you and grow with your work, so locking them at install time would force the foundation to guess your domain.

The 25-value cap is structural, not aspirational

Distinct values across all user-facing dimensions stay under 25. An advisory fires at 80% (≥ 20 active values), surfacing consolidation candidates; raising the cap needs a deliberate tagging_cap_override with a written reason — the system denies a silent widening. The ceiling is the working-memory limit on faceted vocabularies: past it, decision fatigue produces variant-creation rather than canonical-term selection, the vocabulary fragments, and your own recall collapses. The number isn't arbitrary; it's the point where a controlled vocabulary stops being one.

One more coupling: the folder-mirrors-tag invariant. A file in a structural cluster folder carries the cluster's #dimension/value tag, just as it carries the matching lineage field. The frontmatter pillar enforces the field side; tagging enforces the tag side; together they keep the folder hierarchy and the taxonomy aligned, so moving a file across clusters moves both.

How it operates — the write-time sequence

  1. Grammar. Every tag must match the hierarchical regex; failures are denied.
  2. Dimension allowlist. Every dimension must be in the merged allowlist; unknowns are denied with a registration suggestion.
  3. System-utility canonical-value check. Tags under #log/* or #status/* must match a registry value; near-matches are denied.
  4. Tag presence. A non-exempt file with no tag surfaces an advisory — it flags a potential graph orphan at capture, and never blocks.
  5. Cap watch. The librarian counts distinct user-facing values at audit time; 80% usage surfaces a consolidation prompt.

Naming — filenames and paths as contract

The gap in native Claude Code

Claude Code reads whatever paths you hand it and imposes no filename or path discipline; plan mode creates artifacts with no naming contract. So filenames drift, slugs come in mixed shapes that consumers can't normalize, operational exhaust leaks into commit history, and the vault's first-level shape stops being walker-discoverable. The naming pillar makes the path itself part of the contract.

The system reads filenames and paths on every write — to decide whether a file belongs where it lives, to route plan-tree artifacts, and to keep exhaust out of git. The rules are quiet until a write would drift the structure, then they surface the drift before it lands.

The vault-root structure and slug grammar

A fresh brain-stem vault is deliberately lean at the top. The scaffolder seeds exactly one real folder — Vault Writers/ — authors the vault-root CLAUDE.md contract, and wires a set of symlink shortcuts to surfaces that live outside the vault. Everything else is yours to create as your work needs it.

At the vault rootWhat it is
Vault Writers/The one seeded folder — the catalog of systems that write into the vault.
CLAUDE.mdThe vault's top-level contract — the one permitted bare file.
Plans/ symlink→ your plan tree (~/.claude-plans).
Skills/ symlink→ your skills directory.
Wiki/ symlink→ the universal Library.
Projects/ symlink→ the project binders.
Work/ symlink→ your deliverable work area (~/work); exempt from the vault's per-folder index mandate — though each project's deliverables/ and reference/ subfolders now get their own _index.md from a separate work-scoped pass.

The naming pillar enforces a documented-root allowlist: a file written at the vault root must sit in one of the documented names (or be the exempt CLAUDE.md), otherwise it draws a placement advisory pointing you to move it or register the folder via /govern register --kind folder. The allowlist additionally tolerates two optional conventional folders — Archive/ and Daily/ — if you keep that practice, but the scaffolder creates neither; they are permitted, not shipped.

Every slug — filename, folder, frontmatter id, plan prefix — follows one shape: [a-z0-9-], lowercase ASCII, digits, and hyphens. No spaces, no underscores, no mixed case, so every consumer can normalize identifiers without per-surface special-casing. The one human-readable root name (Vault Writers/) is a navigation surface — an explicit exception, not a license to deviate inside it.

Gitignore at-depth — a real leak

A gitignore pattern that should match at any depth (Tags/, node_modules/, .DS_Store) needs the **/ prefix. Bare-name patterns match only at repo root; deeper instances slip past — a production-scale bare-name bypass once leaked ~100 operational files into a reference deployment. The rule ships as inline comments in the foundation .gitignore today and promotes to a pre-commit hook if drift recurs.

Plan slug discipline

Plan-tree files under ~/.claude-plans/ carry the strictest naming contract in the vault. Three rules fire at write time:

  • NN- numeric prefix — creation-order, never backfilled; the next plan takes one above the highest existing prefix, and gaps from retired plans stay as gaps.
  • Status marker — a **Status:** bullet, a status: field, or a status key in manifest.json; a plan with none is denied (the index walker would file it under "Unknown").
  • Shame-slug rejection — auto-generated filler like async-wiggling-donut is denied; a plan needs a descriptive slug that names the actual work.

Two sanctioned paths handle all three in one flow: /new-plan (ad-hoc scaffolding of the canonical quartet + manifest) and /backlog-research (the same, research-first). Hand-writing a plan directory usually fails one of the rules; a per-write PLAN_STATUS_OK=1 override exists for the rare deliberate case and lands in the audit log. Sub-task files at depth ≥ 3 additionally carry a parent_plan: field so plan-tree walkers resolve ancestry without parsing paths — drift there is an audit-time finding, not a write-time block, so an in-flight reorganization isn't trapped by the gate.

How it operates — the write-time sequence

  1. Root-or-exempt. A vault-root file must be in a documented root or the single-file exemption set; a new root path surfaces a placement advisory.
  2. Slug grammar. Every slug — filename, folder, id — follows the kebab [a-z0-9-] convention; the only write-time slug checks are the plan-tree gate (next) and the tag-grammar deny, while ordinary filenames carry the convention without a write-time block.
  3. Plan-tree rules. A write under the plan tree checks the prefix, the status marker, the shame-slug regex (these three deny), and the parent_plan: lineage (audit-only).
  4. Path-routing match. A write to a user-defined folder validates against your overlay's path_routing rule — lineage fields, tags, tier.

File-type contracts — the body schema

The gap in native Claude Code

The platform's write hook can allow or deny a save, but it carries no idea of a per-type body shape — no size caps, no required sections, and no notion that some files are create-only (must never be overwritten) or append-only (must never be replaced wholesale). So a routine regeneration can silently clobber a generated artifact, and there's no structural backbone a sweep can safely regenerate against. File-type contracts supply the body rules.

Frontmatter governs the YAML at the top of a file; file-type contracts govern the body — how long it may be, which sections must be present, and what the system may do to it at write time.

The matcher / parameter split

A contract has two pieces in two pillars. The matcher declares which writes trigger this contract (it lives in a sibling pillar — mandatory-files for vault mandates, plans for plan-tree files). The contract declares what the body must look like: write_shape, size_limits, required_sections, prohibited_sections, frontmatter_enums, and any structured-body rules. The split lets one contract be reused across matchers and lets a matcher swap parameters without rewriting the contract — the same separation Kubernetes draws between ValidatingAdmissionPolicy and its paramKind. And because the rules live in data (the contract) rather than in the hook script, tuning a size cap is a data change, not a code change.

The write-shape enum

Every contract carries a write_shape — the single most protective field, because it's what stops a regeneration from destroying a generated file.

ShapeWhat it permitsDefault class of files
create-onlyInitial write succeeds; later overwrites are denied. Edits go through the file's amender.Generated registration records: writer descriptors, ADRs, ideation briefs, the vault-root CLAUDE.md.
replaceFull-file replacement on every write; prior content is snapshotted for diff-review.Files regenerated as a unit: _index.md, manifest.json, tasks.md, deliverables, Library articles.
append-templateWrites must append a templated block; the contract carries the template and any sentinel markers.Append-only records: handoff.md, spec.md sub-sections.
prepend-row-newest-firstWrites prepend a new row to a sentinel-fenced table, newest first; the surrounding prose is untouched.The episodic chronicle (the memory log).

Overlay collisions and the override discipline

Because the overlay is union-read, it can shadow a foundation contract. The system permits it — but every shadowing entry must carry an _override_reason. On each overlay write a collision check runs; a shadow lacking a stated reason is denied with the offending entries enumerated. A per-write escape hatch (--force-override, or R52_FORCE_OVERRIDE=1 for direct edits) exists for bootstrapping or bulk import, but neither persists — the next write goes back through the gate. The discipline serves two ends: the reason travels with the divergence as a permanent audit trail, and the explicit bypass keeps casual overrides from accumulating silently.

How it operates — the contract checks

Unlike the first three pillars, the body contract is not a write-time block. The matcher resolves which contract a path triggers, and the body rules below are reconciled by the librarian's coverage audits at session close; the one piece checked at the write is the frontmatter coupling.

  1. Matcher resolution. The matcher pillar names which contract this path triggers; no matcher → no body contract applies (the file still validates against the type registry at write time).
  2. Write-shape. The declared shape records what the file permits — create-only forbids overwrites, append-template forbids wholesale replacement, replace regenerates as a unit. A write that violates the shape surfaces as audit drift; create-only files route their edits through the file's amender by convention.
  3. Size limits. A declared size_limits.max_lines is an audit-time check, not a write-time block — the runtime's only synchronous line cap is a single hardcoded hub-file guard.
  4. Required / prohibited sections. The contract's required and prohibited headers are reconciled by the coverage walk; the runtime ships no write-time body-structure guard.
  5. Frontmatter coupling. Contract-declared frontmatter requirements and frontmatter_enums are the one body-contract layer checked at the write — a vault-writer whose writer_kind or status falls outside the enum is denied by the guard; other types' contract enums are reconciled at audit time.

How the four pillars check one write

A single write to one vault file is checked against all four pillars in sequence — type, then fields, then placement, then tags, then body. The strict-tier checks deny; the rest teach.

Why this design

ChoiceThe reasoning
A closed vocabulary per type, declared at write timeAd-hoc per-file fields fragment the field set and no consumer reads them. A registered vocabulary keeps every field queryable.
Enforce at capture, not "later"The cost to add a field is lowest at the write. "Later" defers indefinitely and loses the lineage permanently.
A faceted #dimension/value grammar with a hard capFlat tag namespaces fragment; the 25-value ceiling matches the working-memory limit on faceted vocabularies.
Kebab-case everywhereConsumers normalize on one shape; mixed-case and underscore slugs break equality checks and leak into URL-like surfaces.

Every choice here is anchored to either a platform constraint or a convergent pattern unrelated fields rediscovered — never to taste. The full per-decision evidence, with citations, is on the research rationale.
