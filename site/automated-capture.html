Automated capture

Connectors and writers bring outside information — emails, meeting transcripts, calendar data — into your vault, structured and reusable, without you re-pasting anything into every session.

Two big ideas: structure it deterministically so repetitive processing stays cheap, and keep everything local so you are never penalised for changing your mind.

Two ideas that anchor the whole system

Determinism first

The default posture for a capture pipeline is mechanical: parse the source, apply a fixed transform, emit a content-addressed packet, write once. No model call needed. That posture makes repeated processing cheap and trustworthy — run the same connector ten times and get the same result. When a judgment call genuinely requires language understanding (audience sentiment, action-item attribution), you opt in to an LLM lane; the rest of the pipeline stays mechanical.

This posture is built into the architecture — it is not just a recommendation in the docs. But it has practical consequences: the pipeline enforces a single writer-reconciler (one process owns every write the pipeline makes), deduplicates by content hash, and routes the LLM step before the write — so even the judgment-call lane produces a deterministic packet that the reconciler applies.

Flexibility — never taxed for changing your mind

Raw source data, per-day processed outputs, the append-only SQLite manifest, and any structured outputs (markdown notes, JSON, queryable SQLite tables) are all kept locally and permanently. The only ephemeral piece is the in-flight staging packet — and once it is applied, the manifest records exactly what was written and when.

That means you can start with action items from email, add a second source later, and the system already has the structured data to build from. If you want a dashboard tomorrow, the SQLite the pipeline built is already there. Cross-source deduplication works when you route multiple sources into one keyed destination — that is a routing decision you configure, not something the system does automatically.

What ships vs what you build

The substrate ships: registration, the staging-packet format, the writer-reconciler, the append-only manifest, fan-in/fan-out routing, and both the deterministic and LLM-amender write lanes. Connectors are user-built — a connector is a script that pulls from a source (Granola, Gmail, or anything else) and emits a packet. The registration system records granola and gmail as valid labels, but the connector logic itself is yours to write.

Scheduling capture runs

Capture connectors run on whatever schedule you set. The two turnkey scheduled jobs in brain-stem are the writer-reconciler and the doc-amender; everything else is a hand-rolled launchd plist calling dispatch.sh. For how the orchestration layer handles scheduling, headless runs, and the propose-and-gate flow, see Orchestration & scheduling.
