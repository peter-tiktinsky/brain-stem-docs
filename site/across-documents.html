System governance · the cross-file pillars

Across documents & folders

The second family of governance rules sees more than one file at once — what a folder must contain, and which documents move together. Two pillars: mandatory files and doc-dependencies. Both answer to the same hard platform fact — a write-time hook sees exactly one file — so both lean on advise plus an audit-time backstop. This page takes each in turn: the gap it fills, why it's shaped this way, and how it operates.

The narrative companion to the foundation's mandatory-files and doc-dependency rules. Where the single-document pillars govern a file in isolation, these two govern the relationships between files.

Mandatory files — nothing is left unlabeled

The gap in native Claude Code

Claude Code auto-loads exactly one CLAUDE.md at session start, and otherwise reads only the files you point it at. It has no per-folder index discipline — so a folder is an opaque leaf: to learn what's in it, Claude must read every child file, and past about ten files that becomes token-prohibitive. Nothing guarantees the vault's contract documents are even present. The mandatory-files pillar makes the orientation documents exist by construction and stay current without anyone tending them.

A mandatory file is one the foundation declares must exist. The vault has two mandates — one at vault root (CLAUDE.md) and one per folder (_index.md) — and both are structural: the system creates and maintains them on your behalf rather than depending on you to remember. The intent is that the contract documents are present by construction; you never hit a missing-index advisory on a folder you wrote to last week.

The two mandates

MandateWhat it isHow it's kept
CLAUDE.md at vault rootThe vault's top-level contract — the first document Claude reads, naming the operational conventions, surfacing the spokes, and orienting any session.A create-only file-type contract; your edits are honored through the contract's amender, never a wholesale overwrite.
_index.md per folderThe folder's hub — one-read orientation for Claude, and a navigation table of contents for you.Bootstrapped and reconciled by the librarian's index-maintain sweep across general vault folders; under a registered Work surface, the live post-write hook also bootstraps one on the first write into that folder.

The per-folder mandate exists because of a hard cost curve: past about ten child files, walking a folder file-by-file becomes prohibitively expensive in tokens; past about fifty it stops being feasible. The _index.md collapses that to a single read — and it serves two consumers at once: Claude reads it for orientation, you read it as the folder's table of contents. Its body shape is itself a file-type contract: an H1 matching the folder name, a 2–4 sentence context paragraph, and a File | Lines | Type | Description contents-enum table wrapped in sentinel markers. That contract is the structural backbone the maintenance layers regenerate against.

The exempt set, and extending it

The exempt set is a positive list — only enumerated paths are exempt; any other folder carries the mandate. Foundation ships exactly one exemption:

Exempt pathWhy
Work/**An external-root symlink surface (your deliverable work area, outside the vault root) handed to dedicated Work-scoped machinery rather than the general vault sweep. The live post-write hook bootstraps an _index.md on the first write into a registered Work subtree, and a dedicated Work-scoped pass (work-index-maintain) mints one inside the deliverables/ and reference/ folders under each work project; the project roots and the ~/work/ root itself stay index-free. The vault-wide index-maintain sweep skips Work/** by default, de-exempting only the Work subdirs you've registered.

If you have a folder whose contents are machine-emitted or append-only, add it as a path glob to your overlay's mandatory_files exemption list; the next matching write then proceeds without bootstrapping an index. (It's a deliberate, lockstepped change — the same glob is mirrored into the post-write hook and the index sweep so all three agree.) Adding a new documented root is a different, heavier operation — the naming pillar's six-item new-root checklist, which couples the root to its spoke, type, and librarian coverage in one atomic commit. See The single document.)

How it operates — the three-tier maintenance architecture

The mandate is upheld by three independent layers that converge on one property: every non-exempt folder has an _index.md, and every _index.md reflects its folder's current contents. No single layer closes the gap alone — that redundancy is the design.

1

Live, on a Work write

A freshly-landed deliverable's folder is indexed the moment it's written.

HowA post-write hook fires after every Edit / Write, but acts only when the write lands inside a registered Work subtree: on the first write into such a folder it bootstraps the missing _index.md. Writes to general vault folders are left to the sweep; if the folder already has an index the hook short-circuits, so it never recurses on its own emissions.

2

Periodic sweep

What the live hook missed still gets caught.

HowA librarian capability walks the vault on /librarian full and on a schedule — hand-created folders never written to, files added by direct filesystem edits, row-ordering drift. Mechanical: it fixes what's derivable, flags what isn't.

3

Deep audit

Author intent is reviewed, never silently rewritten.

HowThe same capability run --deep emits an opt-in semantic report on description quality and ordering. It's the only layer that touches meaning, so it stays opt-in.

The mechanical / semantic boundary — the self-healing contract

The system mutates values it can derive without judgment — file presence, line counts, type fields. Anything that requires reading what a file means — its description, its position in the contents table, whether a path should be exempt — surfaces as a finding for you to review rather than being auto-applied. That boundary is what lets a machine keep the indexes current without ever eating the prose you wrote around them. It's also why an index "can't go stale" without being either fixed or flagged: staleness is a derivable property.

How it operates — the write-time sequence

  1. Work-surface gate. The hook acts only when a Work surface is configured and the write lands physically under it ($WORK_HOME/ or Work/ in the vault); a write to any other folder no-ops and is left to the sweep.
  2. Index-presence short-circuit. If the folder already has an _index.md, the hook exits — which is also what makes it self-exempt, since writing an index leaves the folder already indexed and it never recurses on its own emissions.
  3. Registered-pattern match. The folder's vault-view path is matched against the registered path_routing globs; an unregistered Work subtree matches nothing and gets no index.
  4. Atomic bootstrap. A registered, index-less Work folder gets an _index.md minted against the body contract, one contents-enum row per child by derivable values only.
  5. Audit-trail. The periodic sweep is what bootstraps and reconciles every general vault folder, and catches everything the live hook doesn't cover.

Common anti-patterns

Anti-patternWhat goes wrongBetter path
Hand-authoring _index.md on a new folderThe bootstrap mints the index anyway — the sweep on a general folder, the live hook on a registered Work write — and may race a hand-authored one.Let the bootstrap create it; edit afterward through the amender.
Deleting _index.md to suppress the mandateThe next index-maintain sweep re-bootstraps it (as does the next write inside a registered Work subtree); the sweep also flags the deletion.If the folder is genuinely indexless by design, register the exemption — the waiver lands with rationale.
Editing the contents-enum table by handThe live-sync may overwrite your edits on the next reconcile.The table is mechanical (presence, line counts, types); edit descriptions and ordering through the amender, which preserves them.
Trusting the Tier-1 hook aloneIt misses folders never written to, and direct filesystem edits bypass it entirely.Run the index-maintain sweep periodically — the three tiers are layered for a reason.

Doc-dependencies — content that lives in more than one place

The gap in native Claude Code

The platform's write hook fires once per tool call, and the payload carries exactly one file path. It is structurally single-file-blind — it cannot inspect a multi-file change set, so it can never know that editing one document should imply editing another. Coupled documents drift: one half of a paired doc is updated and the other silently rots. The doc-dependency pillar is how the system makes those couplings visible at the one moment it can — the write.

A doc-dependency is a coupling between files whose content is bound by design — a destination fed by several writers, two docs that mirror the same authoritative content, a record that should track a change elsewhere. The core mechanism is a write-time advisory: at every write the engine checks whether the target appears in the registry and, if it does, injects "this write touches a registered dependency — here are the related files" alongside the write decision. It never blocks. Its job is to put the coupling in front of you at the moment of edit, when the cost to keep both sides synchronized is lowest.

What the pillar tracks — two advisories and one active entry kind

The pillar fires two write-time advisories and recognizes one active kind of registered relationship. None of it ever blocks a save.

MechanismRule / kindWhat it does
Cascade advisoryR-54When an upstream doc declared in the registry is edited, its declared downstream consumers/mirrors should follow — the hook names them in the same session and the session-close audit re-checks. advise
Broken-wikilink advisoryR-48A file written with a [[target]] wikilink to a non-existent target surfaces the broken target(s) at write time, pointing you to repair it or create the file. advise
Writer fan-in entrykind: writer-fan-inThe one active registered kind — a destination fed by several named writers. The doc-amender consults the entry's amendment strategy to merge each emission, so it can actually act for you (see the vault-writer system).

Foundation ships no write-time entries[] by default — those are yours to add to your overlay's doc_dependencies registry as you encounter coupled content. What it does ship is an entity-mirror parity registry covering three relationships that exist in every vault by virtue of the system itself:

EntityCanonical fieldMirrors to
skilla SKILL.md's descriptionits Skills/<name>.md design note (strict) + its Skills/_index.md catalog row (presence-only)
plana plan spec.md's statusits manifest.json top-level status (strict)
memory-filea memory file's descriptionits MEMORY.md index row (presence-only)

These parity edges are checked at audit time, not blocked at write time. Every other relationship in your vault — coupled spokes, an updates ledger, a catalog mirroring a per-row history — is registered in your overlay as you meet the case.

How this relates to writer fan-out — the forward path vs. the safety net

The cleanest way for coupled files to stay in sync is the writer's own fan-out: a writer declares all the files it maintains as destinations and writes them in one run (see the vault-writer system). In the shipped system that is the forward path. The doc-dependency cascade is the thin advisory rail that catches the edges a fan-out doesn't cover — most importantly, a human hand-editing one of the coupled files. Both point at the same property: when one document changes, the documents bound to it don't silently drift. The writer-fan-in entry is the registry's link to that pipeline; the R-54 cascade advisory and the entity-mirror parity registry cover the rest.

How it operates — the write-time sequence

  1. Registry match. The engine reads the merged registry and checks whether the target matches a registered entries[] relationship — by its primary path, one of its declared mirrors, or a primary_dir prefix. The entity-mirror edges (a SKILL.md, a plan spec.md, a memory file) are not part of this write-time match — their parity is checked at audit time, not here.
  2. Advisory composition. Any matched entries[] relationship composes the cascade advisory naming the related files. Acting for you on a writer-fan-in entry — the doc-amender merging each emission by the entry's registered strategy — is a separate pipeline run, not something the write hook dispatches.
  3. Advisory emission. If any match fired, the advisory rides along with the write decision — the write proceeds, but the related-files list now travels with the session.
  4. Session-close audit. The librarian re-checks every advisory the session surfaced; if a related file went un-touched, it records a cascade-pending finding for review.

When you want to proceed without touching the related file, a cascade waiver (cascade_waiver_write <entry_id> "<reason>") records why the divergence is intentional, and the waiver lands in the run log. Waiving is the right move — deleting the entry to silence the prompt loses the relationship permanently and strips the audit trail.

Common anti-patterns

Anti-patternWhat goes wrongBetter path
Registering every duplicate as a dependencyThe advisory loop becomes noise; you train yourself to dismiss it.Register only content that's authoritatively duplicated by design — casual copy-paste doesn't belong in the registry.
Adding a dependency without the override disciplineAn entry that shadows a shipped edge without an _override_reason is denied by the collision gate; a duplicate id collides with an existing edge.Add the entry to the overlay's doc_dependencies slot with a unique id, and a stated _override_reason on anything that shadows a shipped edge.
Deleting an entry instead of waiving the cascadeThe registry no longer reflects the relationship; future writes lose the advisory permanently.File a waiver — the registry stays accurate and the next genuine cascade still fires.
Labeling a plain mirror as writer-fan-inThe doc-amender tries to merge a file that no named writer emits to; the merge produces nonsense.Only a destination that receives emissions from named writers is writer-fan-in; a hand-maintained mirror is a cascade edge, not a fan-in.

Why write-time can't see a cascade

Single-file-per-call blindness is a structural fact, not a design compromise

Both pillars on this page are shaped by one platform property: the pre-write hook fires once per tool call, and the payload carries exactly one file path. It cannot inspect a multi-file change set — so "these files must change together" can only ever be advisory at write time. Real cross-file enforcement happens at two later boundaries: the commit boundary, and a session-close parity audit that re-checks every advisory the session surfaced. This isn't something a future version engineers away; it's a property of where the platform lets a guard stand. The design response is the same in both pillars — advise at the write, and back it with a periodic audit a single-write guard structurally cannot be.

Why this design

ChoiceThe reasoning
Auto-bootstrap the index; never demand itA mandate you have to remember is a mandate that rots. Creating the index automatically makes the contract present by construction.
Regenerate indexes, never hand-tend themA status file kept current by a person rots the moment attention moves on — the documented "second-brain" capture-then-drift pattern. A regenerated index can't go stale.
Three layered maintenance tiersThe live hook misses never-written folders and out-of-band edits; the sweep catches them; the deep audit reviews meaning. No single layer closes the gap.
Sentinel-fence the machine's regionThe sweep replaces only the text between <!-- … --> markers and refuses to write if they're missing, so machine and human prose coexist in one note.
Surface dependencies at the moment of editThe cost to keep two coupled files in sync is lowest while you're already editing one. An advisory there beats discovering the drift weeks later.
Advise at write time; enforce at audit timeA pre-write hook sees one file, so a cross-file rule can't be a write-time block. The session-close parity audit is the backstop a single-write guard structurally cannot be.

The "regenerate, never hand-tend" and "structural, not disciplinary" choices are the categorical difference brain-stem bets on against off-the-shelf second brains. The full case, with citations, is on the research rationale.
