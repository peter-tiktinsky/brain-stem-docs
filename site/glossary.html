Glossary

Every coined word you’ll meet across this site, in plain language. The terms split into three groups by what kind of thing they name: capabilities of Claude Code (the platform brain-stem builds on), pieces of brain-stem (what this system adds on top), and general ideas from knowledge management, AI, and software that the docs lean on.

A reference appendix. Skim a group, or search the page for a single term.

How to read the three groups

Claude Code terms are native to the engine — they exist whether or not you run brain-stem. brain-stem terms are things this layer introduces by wiring those native pieces together. General terms are established concepts from the wider world the design borrows. When a brain-stem idea is built directly on a Claude Code primitive, its definition says so.

Claude Code — the platform underneath

Native to the engine brain-stem runs on. These exist out of the box, before any personalization.

TermWhat it means
Claude CodeThe AI engine brain-stem runs on top of. It owns skills, rules, settings, and memory; brain-stem owns the vault, plans, binders, and the Library.
CLAUDE.mdA guide file Claude Code automatically reads the moment you launch a session from a folder, so it loads that folder’s context without you pasting anything.
context window (working window)The finite amount of conversation the assistant can hold at once before it has to trim.
compactionWhen a long session fills the window and Claude Code summarizes the history, permanently discarding the verbatim detail.
hookA script Claude Code runs automatically at a lifecycle moment (before a write, at session start or end, on each message). brain-stem hangs most of its behavior on these.
PreToolUse hookThe single point where Claude Code lets a script intercept a file write and answer allow or deny. brain-stem adds a third answer — advise.
auto-loaded memory file (the cap)The one memory file Claude Code reads at session start, hard-capped at the first ~25 KB / 200 lines, with everything past it silently dropped.
plan modeClaude Code’s native mode for working out a plan, which brain-stem replaces with a durable plan tree.
skill / slash commandA named capability you trigger by typing a slash command (e.g. /new-plan) in a session.
print mode / headless (claude -p)A non-interactive run that executes one instruction and exits — used for orchestrated and scheduled work.
settings / permissionsGlobal and per-project config that Claude Code merges, where a project’s “deny” overrides a global “allow.”
routines (/schedule, /loop)Claude Code’s own cloud-scheduling features, distinct from brain-stem’s local macOS scheduling.
claude-memAn optional Claude Code plugin (not part of the install) that auto-injects recent observations and is searchable on request.

brain-stem — what this system adds

Pieces brain-stem introduces by wiring Claude Code’s empty extension points into one enforced system.

TermWhat it means
brain-stemThe personalization layer that builds and tends your vault on top of Claude Code, wiring its empty extension points into one enforced system.
the vault (brain)The single plain-Markdown Obsidian folder, named brain, that you browse everything through — mostly shortcuts to real homes elsewhere on disk.
the librarianThe automatic routine Claude runs that quietly tends the vault, rebuilding every generated index, list, and binder page from the real files so they can’t go stale.
write-time governanceThe discipline of checking each file against the conventions at the exact moment it’s written, rather than trusting anyone to remember the rules later.
the governance enginebrain-stem’s rulebook, consulted on every file write inside the vault.
the doormanThe guard script that runs on every write and emits one of three verdicts.
allow / advise / denyThe three write verdicts: allow proceeds silently, advise proceeds but injects a reminder, deny blocks the write. Advise is the everyday workhorse; deny is reserved for structural corruption.
the eight pillarsThe eight slices of policy the rulebook is organized into, spanning the single document, across-documents, and writers-and-lifecycles.
the foundation & the overlayThe sealed, never-edited shipped rulebook (foundation-master.json) plus your separate extension layer (overlay-master.json), deep-merged at every write so upgrades never clobber your additions.
/govern registerThe single audited command (six kinds) that writes every personalization into your overlay or scaffolds a work project. You usually describe what you want and Claude runs it.
_override_reasonThe mandatory free-text note an overlay rule must carry to override a shipped rule; without it the write is denied.
file-type contractThe per-document-type rulebook governing a type’s required fields, allowed values, where it may live, and its body shape and size caps.
compliance tiers (Strict / Standard / Minimal)The per-type setting that decides what a failed contract does: deny the write (Strict, for machine-emitted files), advise at session close (Standard, for your own notes), or skip validation (Minimal).
write-shapeThe contract field that controls how a file may be written (create-only, replace, append-template, prepend-row), so a regeneration can’t silently destroy a generated or append-only file.
mandatory _index.mdThe rule that every non-exempt folder must hold an auto-generated contents index, so a folder is never an opaque pile Claude must read child-by-child.
doc-dependency (cascade advisory)A registered coupling between files that must change together; at write time the engine names the related files (it never blocks) and a session-close audit re-checks them.
survivorshipThe guarantee that your hand-edit always wins over an automated regeneration, detected by a two-signal check (a frontmatter marker plus content-hash drift).
the plan treeThe writable, plain-text store of all long-running work at ~/.claude-plans/, placed outside Claude Code’s guarded config folder and surfaced as Plans/.
the quartetThe four core files every plan holds: spec.md (fixed goal), manifest.json (source of truth), tasks.md (generated checklist), and handoff.md (session diary).
manifest.json (source of truth)The plan’s real task list and status in checkable structured data, recorded so an AI can’t fake completion. Everything else about the plan is generated from it.
handoff.mdThe plan’s add-only session diary, newest entry on top, whose “Next session” line lets a fresh session resume cold.
forgeable checkboxThe failure mode brain-stem designs against: an AI claiming it ticked a box it never actually completed.
completed / verified / closedThe three earned-done stages: completed is a human claim, verified is machine-stamped only by a real validation pass, and closed is structurally refused unless the plan is already verified.
master plan & sub-plansThe opt-in structure for a too-large effort: a coordinating master containing sub-plans that are each a full plan, numbered in execution order, that never talk to each other and only publish status upward.
the idea funnel (inbox / backlog / triage)The cheap-capture pipeline where an idea lands in _inbox/ with no plan number, surfaces in the generated _backlog.md, gets classified (novel / duplicate / overlap / deferred), and only graduates into a numbered plan if it earns it.
ideation briefThe optional “Session 0” file capturing why a plan exists — the original idea, the research, and the go/no-go — written before the spec locks in.
the project binderThe per-project roll-up page (surfaced as Projects/) that auto-gathers every plan naming the same project, so work spread across many plans stays coherent. It indexes the plans, it doesn’t copy them.
hub.md (the cover)The binder’s one hand-maintained, pointer-only home page, scaffolded once from a template and then yours to curate.
situating card (_situating.md)The generated orientation card (plan roster, project status, active focus, latest handoff) that auto-loads at the start of every session in a project so the assistant self-orients before its first reply.
the three generated binder indexesresearch-index.md, decision-log.md, and handoff-chronicle.md, each re-derived by the librarian from all of a project’s plans so they can’t drift.
the Library (Wiki)The permanent, universal reference shelf of short one-idea articles that any project can reuse, surfaced as Wiki/; the opposite of a project’s own research.
the WorkshopThe ephemeral, rule-free scratch bench for in-progress research; never auto-loaded, has no vault view, and is read only when a piece is promoted out of it.
promotion / scrubThe propose-then-apply step that turns a finished project finding into a reusable Library article by stripping its project-specific names and dates, while keeping the unchanged original beside it in _raw/.
the moment-of-need Library checkA quiet message-time check that, when your request looks like research, surfaces a one-line pointer to any Library article you already have, so you reuse instead of redo.
the hot index (MEMORY.md)The small read-replica table of contents auto-loaded at session start; each line points to a per-topic detail file read on demand, keeping the load under the native byte cap.
globalizePromoting a project memory marked scope: global up into the global rules/ folder as a schema-validated rule, by explicit confirmation, never touching the hand-edited CLAUDE.md.
the orchestratorThe propose-and-gate dispatcher that reads a plan’s manifest, walks its tasks in dependency order, and runs each in a fresh print-mode session under human-gated supervision. It never edits the plan’s own files.
HITL / HOTLThe two supervision gates: human-in-the-loop halts irreversible or live-mutation work for explicit approval; human-on-the-loop lets reversible work proceed under watch.
circuit-breakerThe orchestrator safeguard that halts new dispatches after two silent failures in a row (a task reporting success while nothing actually changed).
checkpoint (Session Continuity Block)The small fixed-schema autosave of where a session is (plan, task, next steps), written just before compaction and re-injected at the next session start.
context pressure (the pressure ladder)How full the working window is as a percent, driving an escalating ladder of checkpoint prompts (autosave → nudge → mandate → a stop-gate that refuses to end on a stale checkpoint).
the session registryThe machine-local shared whiteboard each open session registers in, so concurrent sessions can see each other and get warned before two edit the same file.
session close (deregister + integrity sweep)The end-of-session cleanup: an always-on part that removes your session from the registry and runs an integrity pass, plus an explicit /librarian session-close that also offers a backup.
the decision-quality protocolA decision-time nudge that fires when the assistant is about to pose a real fork-in-the-road choice, insisting it first run a research pass (re-read goals, check neighbors, read live state, add one unconsidered option).
the vault-writer systemThe machinery that turns outside information (email, transcripts, scrapes) into structured vault assets on its own, shipping the safe-writing parts while you supply the connectors.
vault-writer (writer)Any registered automated system that saves content into your vault without you typing it; each gets a reference page under Vault Writers/.
writer-reconcilerThe sole program allowed to write a destination file, enforced with an OS lock; it de-duplicates by content hash, picks a winner, and writes atomically so two runs never collide.
connectorA user-built script that fetches and shapes a source, then emits a packet for the pipeline to apply; it never writes a destination file itself.
staging packetThe ephemeral content-addressed JSON file (named by the SHA-256 of its body) that holds captured content, where it should land, and its output type — deleted once applied.
doc-amender (LLM-reasoning lane)The opt-in path where a registered prompt runs a model to merge incoming content intelligently, but only composes a packet — the reconciler still does the actual writing.
fan-in / fan-outFan-in routes several writers into one de-duplicating destination; fan-out lets a single writer update many files in one run, every change logged to an append-only manifest.
the Work folderThe governed home for non-code projects under ~/work (surfaced as Work/), one folder per project launched from inside it, kept outside the vault so rebuilds never touch your deliverables.
workstreamA sub-folder inside a master work project that holds one parallel stream’s deliverables and reference; an organizational unit only, never its own launch point.
shame-slugAn auto-generated filler folder name like async-wiggling-donut, denied at write time because a plan needs a descriptive slug that names the actual work.

General terms

Established ideas from knowledge management, AI, and software that the design borrows or builds on.

TermWhat it means
second brainThe established idea of a personal knowledge store kept beside your work (capture, organize, retrieve); brain-stem’s contrast case, which it folds into the “first brain” instead.
ObsidianThe optional notes app the vault opens in. The vault is plain folders and text and works with or without it.
MarkdownThe plain-text formatting used for every readable file in the vault, so any tool can read it.
frontmatterThe small YAML block at the top of a file that declares what it is (type, tags, status) and that the system reads on every write to decide if it’s well-formed.
symlink (shortcut)A signpost to a folder that actually lives elsewhere on disk; it shows up in the vault for browsing, but the real files stay in their own home.
RAG (retrieval-augmented generation)The common “store it and search it” AI-memory approach whose retrieval accuracy dilutes as the corpus grows; a pattern brain-stem deliberately avoids.
semantic / procedural / episodic memoryThe three memory-science kinds brain-stem adopts: timeless facts, how-to knowledge with its reasoning, and dated records of what happened.
ADR (architecture decision record)A short note recording one decision — what was chosen, the options weighed, and why — that supersedes rather than overwrites the old one. brain-stem uses them for any meaningful choice, not just technical ones.
idempotentA process you can safely run twice with no extra effect — the property that lets concurrent session cleanups never trample each other.
atomic write (atomic rename)Writing to a temp file then renaming it into place, so a file is either fully updated or untouched and two writers never corrupt it.
content hash (SHA-256, content-addressed)Naming or de-duplicating a piece of content by a fingerprint computed from the content itself, so identical inputs collapse to one.
deduplicationCollapsing the same captured item arriving more than once into a single record, done by content hash and keyed destinations.
schema (schema-validated)A declared contract for a file’s fields that the system checks a write against before accepting it.
sentinel markersHidden marker lines that fence off the region of a file a machine regenerates, so generated content and your hand-written prose coexist in one file.
kebab-case / slugA lowercase identifier of letters, digits, and hyphens (no spaces or mixed case) used for every filename and id so consumers can normalize them.
fail-openDesigning the guard to degrade toward letting you work (allowing the write) when something is missing or corrupt, never toward locking you out.
launchdmacOS’s built-in scheduler, which brain-stem uses to run capture jobs and timed dispatches locally.
SQLiteA lightweight database kept as a single file on disk — one of the shapes captured content can land in for later querying.
