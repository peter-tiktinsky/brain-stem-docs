Project management · the binder

The project binder

Every project accumulates more than research. As you work, a project's findings, decisions, and session history pile up across all of its plans — and the project binder is the one place that gathers them into a single, always-current view — anchored by a curated home page, with an auto-generated orientation card that loads the moment you start work in that project.

A per-project roll-up · it indexes the project's plans, it doesn't hold their contents. Surfaced in your notes app as Projects/.

The ring binder for one project

A binder is the roll-up of every plan that belongs to a project — its research, its decisions, its handoffs — gathered into one view, with sub-plans grouped underneath their master by lineage. Membership is automatic: each plan's control file names the project it belongs to, so every plan naming this project is gathered in, with no roster to maintain by hand. The binder doesn't copy the plans' contents — it indexes them, linking back to the canonical plan folders.

How a project is identified — the launch directory

All of this depends on one question: which project am I in? brain-stem answers it the simplest way — a project is the directory you launched Claude from. Start Claude Code inside a folder and that folder is the project's identity; both project-scoped surfaces — this project's memory and this project's binder — are selected by it, every session. There is no list of projects to maintain.

Project kindLaunched fromBinder gathers
Codinga code directory — e.g. ~/Code/widgetlibresearch, decisions, handoffs (the code itself stays in git)
Non-codinga Work-area folder — e.g. ~/work/acme-rebrandresearch, decisions, handoffs — plus a pointer to the deliverables it produced
Why the launch directory, not a declared list

A hand-maintained roster of "what counts as this project" drifts out of date the moment you forget to update it. Reading the project from where you launched means there is nothing to keep in sync — the identity is read fresh from reality each session. (The full mechanics of what a launch resolves are on What loads each session.)

What's inside the binder

A binder holds a small, fixed set of pages: one generated orientation card that loads the moment a session opens, one hand-curated cover, and three generated roll-ups re-derived from the project's plans. The orientation card is the one page loaded at session start; the cover and the three indices are read on demand by following their pointers.

hub.md — the cover page (hand-maintained)the project's home page and identity: a pointer-only map that brings the whole project into view without loading its contents. The one binder page you curate by hand; opened on demand when you want the curated project view.
_situating.md · generated · loaded each sessionThe binder's orientation card: the project's plan roster, its overall status, the active focus (the in-progress plan and its current task), and the latest handoff headline — derived entirely from the plans and loaded into every session the instant it opens, so the assistant self-orients before its first reply.
research-index.md · generatedEvery plan's findings rolled into one index, organized by plan, with a link straight into each plan's _research/ folder.
decision-log.md · generatedEvery plan's decisions in one cross-plan list; superseded records are forward-linked, never dropped — so settled choices aren't re-litigated.
handoff-chronicle.md · generatedThe session-handoff records across the project, newest-first, each carrying its Next session: line — so any session resumes cleanly.

The binder's own pages are the cover, the generated orientation card, and three generated indices. The cover additionally points outward — to the universal Library articles this project drew on (the Wiki/ view) and, for non-coding work, to its deliverables (the Work/ view) — those are surfaces the cover links to, not binder pages.

Generated, not tended — how the binder stays correct

The split between the curated cover and the generated pages — the orientation card and the three indices — is deliberate. The cover (hub.md) is scaffolded automatically from a template the moment you register or create the project — and never again; the harness will not overwrite a hub you have already curated. After that first scaffold it is yours to hand-maintain — it is the project's net-new home page, the one surface a person curates. It is pointer-only by contract: it carries links, never imported bodies, so loading it stays cheap.

The three indices — and the orientation card — are the opposite: nobody hand-tends them. A librarian pass re-derives each one from the project's plans — the research index from every plan's _research/, the decision log from every plan's decision ledger, the handoff chronicle from every plan's session diary. These pages are re-derived as part of every session-close, so the binder is refreshed at the end of each session with no manual trigger. Re-deriving them means they cannot go stale: a status file kept current by a person rots the moment attention moves on, but a generated index is correct on every pass with no upkeep.

A reading surface, not a system dependency

Because the generated pages — the three indices and the orientation card — re-derive from the plans (the canonical content stays in the plan folders), the binder is regenerable, not a fragile dependency: delete a generated page and the next librarian pass rebuilds it. The full reasoning — why auto-generated surfaces survive while hand-maintained ones rot — is on the research rationale.

Research is the one facet that travels

Most of the binder stays inside the project: the cover, the orientation card, the decision log, and the handoff chronicle are this project's own record and belong nowhere else. Research is different. A project's findings are often reusable, so research is the one facet that can promote up out of the project and into the universal Library — scrubbed of project specifics so it becomes reference any future project can draw on.

Project researchfindings rolled into the binder's research-index.md
›
Promote & scrubstrip the project specifics; preserve the raw source
›
Universal Libraryreference any project can reuse — the Wiki/ view

The binder is project context; the Library is universal context. How the promotion works — the scrub, the propose-then-apply gate, the immutable raw copy — is covered in full on Dual context layers. The decision log and handoff chronicle have no such path; they stay with the project.

Where it lives

Binders live in the plan area, not inside your notes vault and not inside the folder Claude Code itself occupies — so the assistant can always write to them reliably, and they never get caught in a cloud-sync boundary. Your notes app sees a read-only Projects/ view.
