Write-time governance

The governance engine — core doorman decisions

Claude Code can write files on your machine. The instant it can, something has to decide whether each write is a good idea. The governance engine is that something: a rulebook consulted every single time a file is created or edited, returning allow, deny, or allow-but-advise. The platform exposes exactly one place to intervene before a write lands (the PreToolUse hook), so the shape — a doorman on that single point — is forced. The teach-gently, block-rarely posture is what unrelated policy systems converged on independently.

Enforce every rule in a write-time hook Standing rules in a prose preferences file Prose rules degrade over a handful of sessions and get rationalized away under pressure #33603 #56393 — a documented platform failure, not a hunch. A hook fires regardless of whether the assistant is cooperating. constraint
Intercept before the write (PreToolUse) Checking after the write (PostToolUse) An after-the-fact check can't prevent a bad write and can't even inject a correction back into the conversation #18427. The pre-write hook is the only point that can stop or shape a write — the complete-mediation principle Saltzer & Schroeder. constraint
A third verdict — advise The platform's binary allow / deny CC hooks Allow-or-block alone can't teach, and most policy should be taught. Advise is "allow, plus a reminder"; the write is never stopped. A missing tag earns a nudge, not a block. convergence
Advise by default; reserve hard denial for structural corruption Block on every imperfection (deny-first) Deny-first is hostile and trains people to disable the guard. Deny is kept for the few writes that would corrupt structure (an unknown document type, a malformed required field). The warn-then-block ladder is what OPA Gatekeeper, K8s Pod Security, and ESLint each reached independently. convergence
Let rules earn the right to get strict (data-gated) Asserting strictness on a hunch An advisory is promoted to a hard block only once real usage shows it rarely false-alarms — so it never retroactively denies older work. Matches the monitor-before-you-escalate practice of graduated policy engines OPA Gatekeeper K8s Pod Security. convergence
Fail open — a broken guard never locks you out Fail-closed (block on self-failure) A missing rulebook allows the write; a corrupt overlay falls back to foundation-only. Defensible because the engine governs content, not access authzed. It deliberately pulls against Saltzer & Schroeder fail-safe-defaults — an honest, domain-scoped exception (there is no fail-closed mode today). design doctrine
Three tiers — schema, guard, safety-net — at deliberately different breadth Treating "the JSON schema" as the thing enforced at write-time The schema is the broad authoring contract; the pre-write guard is the narrow authoritative block; the post-write verifier is the advisory-but-complete net. A single hook sees one file before it lands CC hooks, so full-shape conformance is too broad to block on without being hostile. constraint
Keep rules as data in the bundle, read at runtime Hardcoding limits and enums in the hook body A limit — the MEMORY.md index cap, say — lives in a contract the guard reads, so tuning is a data change, not a code change. Hardcoded limits and enums would force a code edit for every tweak. design judgment
Pair every hand-curated enforcement list with a gate-independent backstop Trusting deterministic gates alone A guard can be present-but-dead — non-executable and failing on every install, yet still invisible to deterministic gates that all read green. Because a dead guard passes every automated check, completeness assurance has to be gate-independent. design judgment

Vault integrity — write-time guards and survivorship

The vault is a shared notes folder with two authors — a person editing by hand, and an assistant writing on its own. Two specific write-time rules stop the most common collision: a guard that splits advisory from blocking by what's actually at stake, and a survivorship rule that guarantees the person's hand-edits always win.

Scope of this section

Only the two write-time guard rows and the survivorship row are covered here. The writer pipeline decisions (single reconciler, staging packets, content-hash atomicity, the AI merge lane, log placement, sentinel-guarded indexes, writer registration) belong to the automated capture design and are covered on the automated capture rationale page.

Two write-time guards split by posture — advisory date-warning, blocking writer-reference check Hard-blocking every imperfect note the same way Editing an old note is a legitimate judgment call (warn), but a malformed registry entry silently breaks a sweep (refuse). The platform's hook can only advise-or-block CC hooks, and checking every write is the complete-mediation principle Saltzer & Schroeder. constraint convergence
Two-signal survivorship — a human hand-edit always wins Letting the last writer be the final writer, no check Otherwise a routine regeneration silently overwrites a person's edit. A hand-edit is detected by either a newer edit-timestamp or a content-hash mismatch — the last-writer/merge discipline from OT/CRDT literature, guaranteeing the person wins. convergence
