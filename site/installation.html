Install & upgrade guide

This page takes you from nothing to a working brain-stem setup, and is the canonical reference for every path the quickstart skips — installing over existing files, upgrading in place, and what happens to your data across versions. The core path is three moves: install the foundation, run /onboard once, then open your vault.

Nothing here is magic. Installation is a careful, recorded file copy — preview-first, and every consequential step asks before it acts.

Before you begin

You need four things in place. Each is a one-time setup that lives outside brain-stem itself.

Step 1 — Install the foundation

"Installing" means copying brain-stem out of its downloaded folder and into the one directory Claude Code reads from when it starts up — by convention ~/.claude in your home folder. Download the project and run the installer: 

git clone https://github.com/peter-tiktinsky/brain-stem.git
cd brain-stem
bash install.sh | jq .
The safety property: the bare command writes nothing

bash install.sh on its own installs nothing. It prints an action plan — a structured list naming every folder it would create and every directory of files it would copy, with a reason for each — and then exits, having written zero files. This is a dry run: a rehearsal that changes nothing, and it is always safe to run. Piping through jq . formats the plan so you can read it comfortably.

The dry run is safe even on a home that already has an older brain-stem in it — it does not treat your existing files as an error. The plan's file_dispositions field lists each managed file an install would touch and how: replace (take the new version), new-ship (a file that did not exist yet), or sidecar (take the new version, but first save a copy of your edited one alongside it). When you have read the plan and want to proceed, name the install target and re-run the same command with one word added: 

export CLAUDE_HOME=~/.claude
bash install.sh --apply

--apply is the explicit opt-in that turns the rehearsal into the real install. The design makes the safe thing (preview) the default and the consequential thing (writing files) something you ask for by name. --apply requires you to set CLAUDE_HOME explicitly — the installer refuses to guess where to write and hard-stops (exit code 10) if it is unset. As it copies, it also writes a receipt of every file it laid down, with a tamper-evident fingerprint for each, to ~/.claude/governance/foundation-manifest.json. That receipt is what later lets the system detect tampering and lets an uninstall remove only its own files.

Installing into a folder that already has files

If ~/.claude already contains content that did not come from brain-stem, the installer stops and refuses rather than overwrite it. The dry run tells you everything it would need from you in one pass, under a required_overrides field — instead of failing on the first one and making you re-run to discover the next: 

bash install.sh | jq '.required_overrides'

To proceed past those guards on a real install, add the overrides it named. The most common is the overwrite-risk acknowledgement, passed as a flag (no typing prompt — this is what lets the install run unattended): 

export CLAUDE_HOME=~/.claude
bash install.sh --apply --force-install --i-understand-overwrite-risk --backup-dir <path>

--i-understand-overwrite-risk confirms you accept the overwrite; --backup-dir <path> names a folder where the installer first copies anything it is about to overwrite, so nothing is lost. If your plan directory (~/.claude-plans) already contains plans, the plan will also ask for --retrofit-existing — a one-word acknowledgement that you know those plans are there. The installer never touches your plans; the flag just confirms you have seen them. These overrides apply only to a real --apply install — the dry run never needs them, it just reports them.

Step 2 — Run /onboard

Installing put all the moving parts in place, but the system still does not know you. Onboarding is the one-time guided setup that personalizes everything. Start a Claude Code session and type one command: 

/onboard

A short two-part interview follows: Section A is a confirmation card of facts it already looked up (your name and email from git, your timezone, a proposed vault location) — accept or correct, no typing required; Section B, in your own words, is who you are and how you want the assistant to work. From your answers it writes a settings file, your personal ~/.claude/CLAUDE.md preferences, and your vault. The full walkthrough — every output, how your vault is built, the optional add-ons, and how setup protects your files from a careless second run — is on Onboarding.

Step 3 — Open your vault

When onboarding finishes building your vault, it prints a clear next action:

The hand-off

Open it in Obsidian (Open folder as vault) → select your vault path. Confirm when done.

Obsidian is a free note-taking app that treats a folder of plain-text notes as a connected knowledge base; brain-stem uses it as the human-facing window into your vault. (You do not strictly need it — the vault is just plain files — but it is the intended way to browse it.) Open the folder onboarding built, and you are set up.

Upgrading an existing install

If brain-stem is already installed and a newer version has been released, you do not uninstall and start over. You upgrade in place — with the same entrypoint you used to install. There is no separate upgrade command to learn.

1 — Get the new source. Go to the folder you cloned earlier and pull the latest code:

cd brain-stem
git pull
If git pull reports "divergent branches" or a "(forced update)"

An older clone may not be able to fast-forward if the published history has since been rewritten. Realign it to the published history — this discards any local commits on your clone (you have no reason to have any) but never touches your installed ~/.claude or your vault:

git fetch origin
git reset --hard origin/main

(Or simply delete the folder and git clone again.) This is a one-time fixup; pulls after it are ordinary fast-forwards.

2 — Preview the upgrade (writes nothing). Run the installer as a dry run:

bash install.sh | jq .

On an existing install this prints a write-free upgrade plan: file_dispositions lists every managed file the upgrade would touch and what it would do — replace, new-ship, or sidecar (take the new version, but first save a copy of any file you had edited alongside it as <file>.foundation-local). The preview exits cleanly and changes nothing, even if your current files are older than the new ones.

3 — Apply the upgrade. When the plan looks right, set the target and re-run with --apply. Because an upgrade replaces your merged settings.json, pass --backup-dir so the installer first saves whatever it is about to overwrite (it hard-stops with exit code 53 if a replace is pending and no backup directory is given):

export CLAUDE_HOME=~/.claude
bash install.sh --apply --backup-dir ~/.claude-upgrade-backup

This delivers every changed managed file to the new version — including whole shipped directories (the skills, the orchestrator, the installer support files, the vault seed) that earlier versions could miss on an older install. Files you edited yourself are never lost: each is updated with your copy saved alongside as <file>.foundation-local, so your changes are preserved and clearly flagged.

It fails loud, not silent — and re-running fixes it

Before recording an upgrade as finished, the installer verifies that every managed file it shipped actually reached the new version. If any file is still stale, it stops with a non-zero exit code (56 — "delivery shortfall"), writes no completion stamp, and leaves the upgrade marked unfinished. There is nothing to clean up: run bash install.sh --apply again and it picks up where it left off and converges. A short, recoverable failure is by design always preferred over a silent, incomplete "success."

One thing the preview does not relax on a real apply

The preview is relaxed about your files differing from the shipped version — that difference is the whole point of an upgrade. A real --apply is stricter: if it detects that brain-stem's own files were edited in place, it still refuses to overwrite them silently and asks you to confirm by passing --force-install and typing the I-UNDERSTAND-OVERWRITE-RISK phrase (exactly as on a first install into a folder that already has files). The relaxed behavior is preview-only.

Version migrations — what happens to your data

Some releases change where or how your data is stored. When they do, the upgrade moves your existing data into the new shape automatically during --apply — you never hand-migrate anything, and nothing is deleted.

The general rule: a migration preserves your data and moves it for you. If a release introduces one, its release notes will say exactly what moved, and the upgrade will carry it out during --apply — so an upgrade across any version holds no surprises.

Removing brain-stem instead

If you ever want to remove it, run bash uninstall.sh from your cloned brain-stem folder with CLAUDE_HOME set (the uninstaller is not copied into ~/.claude). It backs everything up first, then removes only the untouched files it originally installed — any file you edited, and your vault, are preserved. The full story is on Uninstalling and Packaging & runtime.
