Session management

A session is a single conversation with Claude, from open to close — and its working memory is perishable. brain-stem wraps the session in machinery that saves your place as the window fills, keeps concurrent sessions out of each other's way, and reconciles what changed on the way out. This page is the lifecycle; each piece has its own page.

Most of it is forced by one platform constraint — a finite working window that compacts lossily; the rest answers a second gap: a session that starts knowing nothing about its project.

What Claude Code gives you — and what's missing

Natively, the assistant has a fixed-size working window. When a long session fills it, the platform compacts the conversation — replacing the verbatim history with a summary, after which the full detail is gone. The platform also exposes lifecycle moments (session start, before-compaction, stop, session end) where a script can run.

The session lifecycle

Read the arc start to finish. Each moment below is a hook the platform fires; brain-stem hangs one behavior on each. The order matters — saving must happen at compaction, but restoring can only happen at the next start (the platform forbids injecting context at the compaction moment).

Startregister the session; auto-load the project's situating card; on a post-compaction restart, restore the saved checkpoint
›
Each messageread context pressure; nudge you to checkpoint; warn if a peer touched your file
›
Stop / compactionrefuse to stop on a stale checkpoint; save your place just before detail is trimmed
›
Closemark the session done and reconcile what changed
MomentWhat fires
Session startRegister the session in the coordination registry; if this is a restart after compaction, read the saved checkpoint back in and rotate the old one to a dated archive. Also, for any session launched from a registered project (a coding repo or a non-coding Work project), an always-on hook auto-loads the project's situating card as context before the first turn — so the session already knows what the project is and where things stand.
Each messageCompute context pressure, check checkpoint freshness, emit escalating nudges, and surface peer-session overlap warnings.
Stop (attempting to exit)The stop-gate refuses to let a busy session end on a stale or missing checkpoint.
Before compactionPreserve a fresh rich checkpoint, or mechanically extract a fallback one — no model call.
Session endMark the registry row closed; if a peer left work pending, spawn the reconciler in the background. And unless an explicit close already ran, spawn the librarian's commit-free integrity cleanup detached — so indexes and frontmatter stay honest even when you forget to close.
Explicit close/librarian session-close runs the full cleanup chain (next pages) — including regenerating the project's binder indexes and its situating card, so the next session start auto-loads fresh context.

Always-on, and on-demand

Most session machinery is always-on — it runs every session whether or not you ask. A smaller set is on-demand, invoked by you or fired when the work calls for it (often the moment you produce or change material and governance has to weigh in).

Always-on

Runs every session

Session registration and checkpoint restore; auto-loading the project's situating card on every start; per-message pressure nudges and peer-overlap warnings; the stop-gate; the pre-compaction fallback save; deregistration, background reconciliation, and a detached commit-free integrity cleanup on exit.

On-demand

Invoked by you or by the task

A rich /session-checkpoint on request; the /librarian session-close cleanup; and the decision-quality nudge that fires when you're about to commit to a substantive choice.

Where governance enters a session

A session isn't only about staying alive — it's about producing material safely. The moment a write lands inside the vault, the governance engine checks it (allow · advise · deny), and the writer pipeline serializes it. So session machinery and governance interlock: the session keeps your place and coordinates the work; governance keeps each write well-formed.
