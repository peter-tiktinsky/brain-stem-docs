Why these choices — system personalization

Three pillars underpin brain-stem's personalization story: onboarding (the one-time interview that stamps your identity onto the foundation), governance overlay (the seam that lets you extend or override rules without ever touching the shipped base), and packaging (the install/upgrade/uninstall discipline that keeps your edits safe across version boundaries). Each decision table below states what brain-stem chose, what it chose over, and why. Citations link to the central register on the research rationale page.

Decision depth · pillar #onboarding + #governance overlay/register rows + #packaging lifecycle. The core governance doorman rows live on Why: write-time governance.

How to read these tables

Each row records what brain-stem chose, what it chose it over, why, and the ground the decision rests on. constraint means the platform forces the shape; convergence means unrelated fields independently arrived at the same answer; design judgment means neither applies — the choice is a reasoned bet, not a necessity. The strongest signal is a row that earns both the constraint and convergence labels at once.

Onboarding

The personalization seam

Onboarding is the one-time guided interview you run when you first type /onboard. It asks who you are and how you work, then writes your global preferences file and builds your starter vault — the seam where the identical install everyone downloads becomes your system. The split here is clean: the shape of a good first-run flow (smart-default pre-fill, run-once, validate-before-write, never clobber) is convergent setup-wizard practice; the mechanics that make those guarantees real are forced by documented platform behavior. One scope rule governs everything below — brain-stem ships only the minimum-viable seed; the ambitious 27-question wizard was deliberately cut to a separate add-on.

brain-stem choseoverbecauseground
A one-time guided interview as the personalization seam The same generic install for everyone, or hand-edited config per machine You don't assemble a toolkit; you move into a furnished foundation and personalize it at one known seam. The mechanism matches established setup-wizard practice wizard UX. convergence
The assistant can never start onboarding on its own Default skill behavior, where the model can decide to fire a skill Onboarding writes your personal files and builds your vault — it must never surprise you by running unprompted mid-task. The guard rests on a documented platform setting CC skills; only a human typing /onboard launches it. constraint
Run once, then no-op; redoing it takes an explicit --force Letting setup re-run freely, or self-deleting after a successful run A step that writes irreplaceable personal content must never be cheap to repeat by accident; self-deletion mid-run is racy and breaks the resume path. convergence
A two-part interview — a pre-filled confirmation card, then short optional prose Making you type the mechanical facts, or a heavy 27-question wizard Section A is fact-gathering a probe can do (name/email from git config, timezone from the clock); Section B is the genuinely human input — how you work — that no probe can guess, and it is fully optional. convergence
One AI step — a two-pass extraction of your prose into labeled fields Pure deterministic capture of the raw answer Verbose conversational answers must be consolidated to fit the preferences file's size cap (~four lines per block in a ~60-line file). The structural check confirms the summary lands in the right slots. constraint
Validate first, write atomically — block-and-log, never write-and-hope Writing optimistically and hoping it completes The manifest is schema-checked and both preferences files are scanned for leftover placeholders before anything is written; writes go to a temp file then swap; on any failure it stops and reports, leaving no half-written file. convergence
Author the two CLAUDE.md files fresh; ship no pre-seeded placeholder Shipping a pre-seeded placeholder preferences file in the install A file full of unfilled tokens looks broken to a reader and blocks the genuine first-run fill — getting no file is cleaner than a broken one. The global file loads first, then the more specific one, rather than one silently overriding another CC CLAUDE.md. constraint
A re-run never silently overwrites a file you edited Silently overwriting on re-run If a CLAUDE.md differs from a fresh render, onboarding writes a .new file and shows you the diff, leaving the real file untouched until you decide — your edits are sacred, the same preserve-don't-clobber pattern as chezmoi. convergence
End with shortcuts and an explicit "open it in Obsidian" instruction Silently trailing off once the folder is built Building the folder is invisible; the adopter needs an explicit, concrete next action. The hand-off "hands you the key and tells you which drawer to open first." design judgment
Treat external tools as a soft mandate — probe, recommend, frictionless skip A hard requirement that blocks setup For each optional add-on: a read-only presence probe, then a strong recommendation, the honest reason, the exact command, and a clean "skip and do it later." It records your choice and never blocks wizard UX. convergence

Governance overlay & extension

Sealed master, transparent overlay, sanctioned extension

These three rows are the personalization half of the governance pillar — how you extend or override the shipped rules without ever editing the foundation. The core write-time doorman decisions (pre-write hook, three verdicts, advise-by-default, data-gated strictness) live on Why: write-time governance. What sits here is the seam brain-stem opens for you to reshape governance to your situation: a sealed shipped base you never touch, a transparent overlay you own, and one guided command (/govern register) that covers all 6 extension modes — 5 governable-asset kinds (folder, file-type, tag-extension, writer, doc-amender-prompt) plus the project work-spoke on-ramp.

brain-stem choseoverbecauseground
A sealed master plus a transparent overlay One rulebook everyone hand-edits Your additions go in a separate overlay deep-merged on top; the shipped master is never edited locally. This is the structural guarantee behind taking an upgrade without losing your customizations Kustomize git config CSS cascade. constraint convergence
An override must carry a mandatory free-text _override_reason Letting the local rule win silently Because the overlay wins on a collision, a silent override could shadow a shipped rule unnoticed. brain-stem blocks the merge unless the entry carries an _override_reason — deliberately stricter than its own analogs git config Kustomize, matching policy-as-code waiver practice waiver practice OPA bundles. Extend (new keys only) is frictionless and needs no reason; only a collision triggers the requirement. A per-write --force-override flag exists for non-persistent cases. convergence
One sanctioned extension path — /govern register Hand-editing the overlay file directly The skill walks a propose-then-confirm conversation, validates each field, and appends to an audit log; hand-editing bypasses both. The skill covers all 6 modes: registering a folder type, a file-type contract, a tag extension, a writer, a doc-amender prompt, or a project work-spoke. The hook only proposes (and lets you skip); a human runs the skill CC skills wizard UX. convergence

Packaging & runtime

The install/upgrade/uninstall lifecycle

Claude Code reads one startup folder, ~/.claude, and merges its settings by a fixed rule — and gives you nothing else. No record of what an install put down, no way to tell an old shipped file from one you edited, no safe way to take an install back out. Drop brain-stem into that live folder blind and you risk silently clobbering your own work. This pillar is the discipline that closes the gap, and it is reached on both grounds — forced by the single-folder convention, and convergent with what Homebrew, dpkg, chezmoi, Flyway, rustup, and the XDG spec each arrived at independently. brain-stem then goes one step past the typical package manager: a provably write-free plan before the first write.

brain-stem choseoverbecauseground
A preview-first install — install.sh writes zero files without --apply An installer that writes immediately It runs every check, prints a machine-readable plan of every operation, and exits; writing must be asked for by name. The safe thing is the default, the consequential thing asked for — the check-then-apply gate rustup uses. convergence
Refuse an apply when CLAUDE_HOME is unset; let a dry-run assume it Treating preview and apply symmetrically A preview may assume a target to render a plausible plan; a real write may not. Real files only ever land in a folder you deliberately pointed them at. constraint
A generated, fingerprinted receipt of every shipped file A hand-maintained ship-list A hand list always drifts, and a receipt that lies is worse than none. One walk of the source tree defines both what ships and what the manifest records, so they can't disagree — exactly how Homebrew / Nix / apt-dpkg know what they put down. convergence
An edit-preserving uninstall (fingerprint match — remove; mismatch — keep) A blunt rm -rf of the folder Over time your files live right next to the foundation's, so a blunt delete takes your customizations with it. The manifest is the boundary between managed and unmanaged — preserving changed files is what chezmoi does. convergence
Distinguish drift from tamper on upgrade via per-release baselines A blind copy (clobbers edits) or a blind skip (fails to deliver fixes) A file matching a known previous release is stale shipped code — take the new version; a file matching no release is your edit — take the new version too, but snapshot your edited bytes aside as .foundation-local. The sidecar is dpkg's .dpkg-old; the version-keyed baselines are Flyway/Alembic's high-water-mark. convergence
All-or-nothing writes via a per-file rollback journal The Capistrano whole-tree symlink swap, or a recursive subtree copy You can't atomically swap a directory that holds the user's live, interleaved state Capistrano, and cp -R can't honor per-file rename atomicity — so the work is decomposed to a per-file walk with a reverse-replayable journal. constraint
A delivery gate — a half-install is never recorded as success Writing the completion stamp regardless, or a blanket "every file equals shipped" check A blanket check is unsound — it false-refuses the merged overlay and every edited-and-preserved file. The gate checks only that files meant to update did; if not it exits code 56, stamps nothing, and a re-run converges. A short recoverable failure beats a silent incomplete "success." design judgment
Deep-merge settings with your value winning, behind a deletion gate A wholesale overwrite, or per-hook hand-patches The merge refuses to proceed if it would silently drop any of your existing keys; a reconciler then adds missing foundation hooks without reordering yours. Claude Code reads one settings.json by a documented deep-merge rule CC settings, so the shape is forced. constraint
Keep runtime data outside ~/.claude, split durable vs. ephemeral Keeping working state inside the program folder Separating the installed program from the data it generates means uninstalling or reinstalling never endangers your accumulated work — the XDG durable/ephemeral split the foundation already follows. convergence
Keep coordination state machine-local, not in the synced vault Homing the session registry and locks in the synced Obsidian vault A small, hot, frequently-rewritten JSON plus locks in a synced folder is worst-case input — and the cross-machine guarantee is a phantom, since kill -0 and lockf are local-machine constructs Kleppmann. tmux, systemd, git's index.lock, and even Syncthing keep runtime state local runtime-state precedent. convergence
Fence background jobs to the com.brain-stem.* label namespace A careless cleanup that stops unrelated jobs On uninstall it stops only jobs in its reserved prefix and actively refuses to touch an impersonator, aborting rather than risk stopping something unrelated. Reverse-domain naming is the OS-level launchd convention. convergence
