Mechanics

What loads each session

What the assistant knows the moment a session opens is not fixed — it is assembled from the directory you launched in. The launch directory selects the rules, the permissions, the memory, and the project's situating card, every time.

The launch directory is the single input; everything project-scoped is derived from it.

The launch directory is the project

When you start Claude Code inside a folder, that folder is the project's identity. There is no roster of projects to maintain — the location you started in selects everything project-scoped, read fresh each session.

Project kindLaunched from
Codinga code directory — e.g. ~/Code/widgetlib
Non-codinga folder in the Work area — e.g. ~/work/acme-rebrand (see Project organization)

The four things resolved from where you launch

1 · Rules

Behavioral rules

The global rules always load; the project's own rules — found by walking up from your launch directory — are added on top. (Claude Code's own behavior.)

2 · Permissions

Settings

Global and project settings are merged; a project's "deny" wins even over a global "allow." (Claude Code's own behavior.)

3 · Memory

Project memory

The launch directory maps to a project memory area; its small hot index loads at start, with detail read on demand. (Organized by brain-stem.)

4 · Orientation

The project's situating card

A small generated card — the project's plan roster, status, active focus, and latest handoff — is loaded into the session automatically, so it self-orients before your first message. brain-stem's net-new addition, and the subject of the rest of this page.

What's brain-stem, what's not

The first three — rules, settings, and memory — are Claude Code's own loading behavior; brain-stem rides on them and keeps them organized. The genuinely net-new piece is the fourth: the project's situating card, generated from its plans and loaded automatically at start (the curated cover page, hub.md, is read on demand behind it).

What loads from your project directory

Always launch from a project

brain-stem is built around launching from a project directory — a code repo or a Work-folder project. That's what attaches the session to a project and its knowledge base. Launching from your home folder loads only global context with no project attached, so it isn't how you work day to day.

What orients the session — the situating card

The one project-scoped thing brought into view from the start is a small generated situating card: the project's plan roster, its overall status, the active focus, and the latest handoff headline — derived from the project's plans and loaded into the session automatically, before your first message, so the assistant self-orients with no reading step. It is kept small on purpose, so it costs almost nothing to load. The project's curated home page, hub.md, sits behind it and is read on demand — a map, not a payload: open it when you want the full pointer-only index, and the heavy material behind each pointer is read only when the work reaches for it.

hub.md — the curated cover, read on demandthe project's pointer-only home page; open it to bring the whole project into view without loading its contents
Research indexEvery research note the project has produced, organized by plan — its accumulated findings.
Decision logA running record of the choices made and why — so settled decisions aren't re-litigated.
Handoff chronicleSession-by-session notes of what was done and what's next, newest first — any session resumes cleanly.
Library referencesLinks to the universal Library articles this project drew on or contributed to.
DeliverablesThe project's finished work product (the Work binder) — briefs, docs, and reports for non-coding projects.

The binder these pointers open onto — its full anatomy, the situating card included, and how each page is generated — is its own page: The project binder.

Why bring in a map, not everything

A full reference shelf or every project file loaded into every session would be enormous and mostly irrelevant — and the working window is finite. So access is tiered by cost:

TierWhatWhen
Always-onrules, settings, the small memory index, and the project's situating cardat session start
Moment-of-needa one-line "there may already be reference on this" hintwhen a request looks like research
On-demandthe cover hub.md, research, decisions, handoffs, library articles, deliverableswhen a pointer is followed
Never auto-loadedthe scratch Workshopread only when research is promoted
The throughline

Small, high-value context loads up front; everything heavy waits behind a pointer. The launch directory decides which project's small-and-up-front set you get — and the home page is the bridge from that small set to everything else the project holds.

Reference

  • Global rules + a project's own rules — concatenated behavioral context (native Claude Code loading).
  • Global settings deep-merged with a project's settings — project "deny" wins (native).
  • The project memory area, keyed to the launch directory's git root (else the folder itself).
  • The project's situating card (_situating.md) — a small generated status card (plan roster, aggregate status, active focus, latest handoff headline) loaded into every session automatically at start, so the assistant self-orients with no reading step.
  • The project's home page hub.md — the curated, pointer-only cover, read on demand.
  • Session machinery — memory review, the active plan's goal, peer-session awareness, and the moment-of-need library hint.
