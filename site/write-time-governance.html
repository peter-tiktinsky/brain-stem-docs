Write-time governance · conventions enforced at the moment of every write

Write-time governance

Every file and folder, the moment it is created or edited, is checked against a rulebook of conventions. The point is that files are always discoverable by Claude and findable by you — not because someone remembered to apply the right format, but because the system enforced it at write time.

Conventions applied structurally — at the write — not left to memory or discipline.

Two kinds of governance

brain-stem uses the word "governance" in two distinct places, and it helps to name them cleanly before going further:

This section

Write-time governance

Conventions that apply to every file and folder at the moment of creation: frontmatter schema, naming rules, mandatory indexes, doc-dependency chains. Enforced by a hook that fires on every write. The goal: any file the system touches is structurally legible immediately, without a cleanup pass later.

Project management section

Process governance

Conventions for how work is organized and tracked over time: plans, orchestration, project binders. These govern the lifecycle of work — how you propose, run, gate, and close a piece of work — not the shape of the individual files it produces.

The distinction is practical: if you are looking at a specific file and asking "is this well-formed?" or "why is this folder structured this way?", that is write-time governance. If you are asking "how does this piece of work get planned and run?", that is process governance in the Project management section.

Extending or overriding the rules

The write-time rulebook is sealed in a foundation layer — you do not edit it directly. You extend it (register new file types, folder types, writers) or override a shipped rule (with a stated reason) through the overlay seam. That is a personalization concern, not a write-time concern.
