System personalization · extending the rulebook

Governance personalization

brain-stem ships sealed, but it is built to be shaped to your work. A new folder, a new kind of document, a whole template for a kind of project, an automated feed that keeps a structure current — every personalization lands through one governed, audited seam. The shipped rulebook is never edited, so you can take an upgrade without losing a thing you added.

The seam is the overlay (governance/overlay-master.json); /govern register is the only door into it.

One seam for everything personal

The shipped rulebook — foundation-master.json, the composed eight pillars — is read-only. You never edit it. Your changes live in a separate overlay, and at write time the engine deep-merges foundation + overlay into the live rulebook the guard consults. The overlay has one slot per pillar (plus a small slot for machine-local settings like your timezone), so any kind of rule has a place to go.

You don't hand-edit the overlay either. Every change flows through /govern register, a single locked, schema-validated, audit-logged path — each registration is recorded in an append-only action log. Two guarantees fall out of this shape:

  • Your changes survive an upgrade. Because they live in the overlay and not in the shipped file, you can take a new foundation version and your extensions merge straight back on top — and the system always knows which rules are yours.
  • Overriding a shipped rule requires a stated reason. Adding a brand-new rule just merges in. But an overlay entry that would shadow a foundation rule has to carry an inline _override_reason — a mandatory free-text note, not a cryptographic signature. Without it the write is denied. (The only bypass is a single-use, write-time escape hatch at the guard — it clears one write and one write only; there is no permanent off-switch.)
Why a seam, not edits

Hand-editing the shipped rulebook would make upgrades impossible and blur the line between what you changed and what came in the box. The overlay keeps that line bright: foundation stays pristine and reproducible, your layer stays distinguishable and portable, and a collision can only be resolved with a reason on the record.

Where the rulebook comes from

The eight pillars don't arrive as eight files you wire together. They are composed — at release time — into one sealed bundle, foundation-master.json, with one slot per pillar. The composition is deterministic (same inputs produce a byte-identical output; the version is a content hash that excludes build timestamps). You never build it; the installer ships only the composed artifact, and your overlay merges on top.

The picture is simplest as two sources merging into one: the sealed rulebook that ships, and the overlay you own. They deep-merge at every write into the single live rulebook the guard consults.

The door: /govern register

Every personalization flows through one command, /govern register — though, as on the work folder, you rarely type it yourself: you describe what you want, and Claude runs the right registration. It writes through one locked, validated, audit-logged path, and it has six kinds — five register a governable asset, and the sixth sets up a work project.

--kindRegistersLands in
foldera new top-level vault folder (and any file it must contain)a path-routing rule + mandatory-files; updates the vault map in CLAUDE.md
file-typea new document type and its contract — structure, fields, where it may livethe file-type-contracts slot
tag-extensiona new tag dimension and its valuesthe tagging-taxonomy slot
writera new automated vault-writing systema writer-reference under Vault Writers/
doc-amender-prompta new guided doc-amendment prompt (the opt-in LLM lane)a create-only prompt asset
projecta new ~/work project workspace — the work-folder on-rampthe project scaffold + a vault-view routing rule

A file-type contract is the most common thing you'll register: it governs a type's frontmatter (required fields and allowed values), where files of that type may live and how they're named, and its write shape — optionally its body structure too. Plans are deliberately not a register kind; new plans go through /new-plan. Setting up a work project gets its own walkthrough — see the work-folder guidebook.

How a registration actually runs

No registration writes blindly. Every one moves through the same three beats, and you hold the gate in the middle:

ProposeClaude emits the proposed rule — the exact fields it would add — and flags any collision with a shipped rule. Nothing is written yet.
→
You redlineYou approve or edit it field by field. A change that shadows a shipped rule must carry a stated _override_reason, or it's denied.
→
CommitThe validated change is applied atomically and recorded — one row per affected pillar — in an append-only action log.

You can also skip: decline to register and let the underlying write proceed, with the skip itself recorded in the same log. Registration is an offer at the moment of a write, never a wall.

What makes the path safe

Every overlay change runs through a single locked, schema-validated mutation path. The change is checked against the overlay schema before it lands; it's applied under a lock with an atomic rename, so a multi-rule registration either lands whole or not at all; and every action — committed, rejected, or skipped — appends a row to an immutable action log (governance-action-log.jsonl). On any validation or write failure the registration blocks and logs rather than leaving a half-formed rule behind. New assets are create-only, and the project scaffold refuses to clobber a non-empty directory.

What /govern register deliberately does not do: there is no list, audit, or unregister verb. Inspecting what you've registered and catching drift is the librarian's parity audit — a separate capability. And there is no permanent way to switch off a shipped rule: overriding one takes a reason every time, and the single one-off escape hatch (R52_FORCE_OVERRIDE) clears one write at the guard, not a setting on this command.

One foundation, any user profile

People reach for Claude Code to do drastically different work, and no two setups look alike. A few real shapes:

  • A designer using Claude Code for UI/UX work keeps design assets, component notes, and house style rules together in a project directory.
  • A consultant runs several client engagements at once — each its own project directory, with its own statements of work and correspondence.
  • A lead engineer spans both worlds, juggling coding repositories alongside project-management directories across one Claude setup.

brain-stem doesn't pick a side. It's the common foundation underneath all of these, and it bends to any user profile in two ways: you extend its native capabilities to fit your work, and you layer your own structures on top without disrupting anything that shipped. Both land through the one governed seam you've already met — and nothing you add ever edits the foundation.

Taken together, those additions are your own overlay package — the shape of your work, captured. You build it through four moves:

1 · Your folders

Shape the directory

Add the folders your work actually needs — a folder per client and an SoW/ directory for the consultant, an assets/ library for the designer. brain-stem indexes each one as you go.

2 · Your document types

Teach it your kinds of file

Register the documents you actually work in — a statement of work, a design spec, a status report — each with its own rules for fields, naming, and where it lives. From then on they're governed like anything brain-stem ships.

3 · Your templates

Seed your own starting points

Drop in starting skeletons for those documents — a blank SoW, an empty design brief — that you then own and edit. Nothing is seeded for you by default; this space is reserved for what you bring.

4 · Your reusable package

Bundle it to reuse

Capture the whole setup — folders, rules, and templates — as one package you can re-apply to your next project of the same kind, or hand to a teammate. Your overlay, your profile, portable.

The guarantee: your layer never touches what shipped

Build out a whole profile — new folders, your own document types, your templates, an automated feed — and the only files that change are the ones under your project and in your overlay. No shipped foundation file is modified: not the master rulebook, not a contract, not a skill or a hook. That's exactly what keeps your setup portable and an upgrade safe — your layer merges straight back on top of the next version, every time. A test asserts both halves at once: a real working setup is produced, and the foundation is left byte-for-byte unchanged.

Feeds that fill the structure for you

Personalization isn't only static shape. You can register an automated vault-writer through this same seam (/govern register --kind writer) — a feed that keeps a structure current by writing into it on a schedule. It's the same discipline as every other extension: a registered declaration, governed and audited. The deep treatment of how that pipeline works — staging, the single reconciler, output types, the optional reasoning lane — lives in the vault-writer system.

Reference

  • governance/overlay-master.json — your extension layer; one slot per pillar plus a machine-local settings slot. Written only via /govern register, through a locked, audit-logged mutation path; every action recorded in an append-only log.
  • _override_reason — the mandatory inline reason on any overlay entry that shadows a shipped foundation rule; a shadowing write without it is denied (the single-use, write-time escape hatch aside, there is no permanent bypass). It is a free-text note, not a signature.
  • /govern register — six kinds: folder · file-type · tag-extension · writer · doc-amender-prompt (the five governable assets) plus project (the work-folder on-ramp). Each runs propose → redline → commit (or skip), schema-validated and recorded in an append-only action log. No list / audit / unregister verb — inspection and drift-catching are the librarian's parity audit.
  • The four ways to build your own overlay package — your folders · your document types · your templates · bundled for reuse. Every one lands in your overlay or your project, never the foundation; a test asserts a full setup leaves the shipped foundation byte-unchanged.
  • A vault-writer — registered via /govern register --kind writer; a writer-reference under Vault Writers/ declares its kind, its destinations, and each destination's output type (markdown · JSON · database). See the vault-writer system.
