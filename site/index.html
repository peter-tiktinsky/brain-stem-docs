Introduction

The nervous system for Claude Code

brain-stem is the nervous system for Claude Code: it builds the brain (a governed context & memory layer), optimizes the body (Claude Code's native planning, sessions, governance, and orchestration), and grows the stem that connects them — one integrated system wired into how Claude loads and runs every session, not a second brain bolted on the side. This page is the system tour, end to end; for the case for adopting it, pillar by pillar, see Why brain-stem.

Internal reference · documents brain-stem as shipped.

Brain, stem, body

Out of the box you have a brain with no spinal column: Claude can already read your files, remember, plan, research, and write, but each faculty runs alone, ungoverned, and only when you wire it up by hand. brain-stem carries signals both ways and makes them one coherent system — each faculty tuned against what actually works and wired to the others rather than standing beside them. That is what makes the brain itself smarter and the whole better at supporting you and personalizing to your goals. The map below is what that connected body actually does, end to end.

Obsidian is just the window

brain-stem was designed with Obsidian in mind — but it does not depend on it. The whole "brain" is plain folders and Markdown files; any editor opens them, and the system runs exactly the same whether or not Obsidian is ever installed. Obsidian is simply the most convenient window onto what Claude produces session to session and project to project — not the engine underneath.

The one thing Obsidian uniquely adds

The single Obsidian-specific affordance is the graph view — a visual of your knowledge graph, drawn from the tag taxonomy. But the machinery runs on structured frontmatter — it reads the tags only to keep them healthy for that graph; tags and wiki-links are user-side navigation that degrade to plain text in any other reader. Remove Obsidian and you lose the graph picture — and nothing else. Even the Plans · Wiki · Projects · Work shortcuts are ordinary symlinks that resolve in any file browser.

Portable by design

Minimizing lock-in is a deliberate goal. brain-stem leans almost entirely on a computer's native folders and files, universal Markdown, and a set of documented governing rules — so your knowledge never gets trapped in one app's format, and you can plug and play whatever tools you like around it. What it needs to run is short:

To run brain-stem, all you needWhich means
A MacFor now — the scheduler and session-locking use macOS specifics; Linux and Windows aren't supported yet.
A terminalBuilt into macOS — nothing to install.
Local files & foldersAlready there; no database, no proprietary store.
A Claude Code accountFor now — native versions for other harnesses (e.g. Codex, Antigravity) aren't built yet.
The honest line: portable knowledge, a macOS runtime

Portability here is about your knowledge, not the engine. The running system — its background scheduler and the locking that keeps concurrent sessions from colliding — is built on macOS specifics (launchd and BSD lockf), and the project says so plainly: macOS only, today. The governing rules and the vault scaffold are ordinary bash and travel further in principle, but the shipped system targets a Mac. The distinction is the whole point — the brain it produces is plain text that goes anywhere, even while the engine that produces it still needs a Mac.

Multi-homing — the brain travels

Because the brain is nothing but plain text, it isn't locked to Claude Code. The folder brain-stem builds can serve as a reference or resource for any other AI harness, in whole or in part — or you can point a local LLM at it and treat your accumulated knowledge graph as the feeder for a private, searchable knowledge base you query offline. The value you build up stays yours, and stays portable across whatever systems you choose.
