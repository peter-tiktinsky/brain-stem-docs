Context and memory

Out of the box, Claude Code starts every session blank. brain-stem fixes that by keeping durable knowledge in plain-text files on disk — and there are two kinds of it, answering two different questions. This page is the map that ties them together; the deep dives that follow take each apart.

The organizing model: operational vs contextual knowledge, each at project or universal scope.

Two kinds of durable knowledge

Everything the assistant carries between sessions answers one of two questions — and that split is the whole model:

  • Operational knowledge — how to behave. Your preferences, conventions, workflow rules and the lessons behind them. You author it. It is the operating governance you set.
  • Contextual knowledge — what the work is and where it stands. The reference that frames a topic, and the record of what's been done. It accumulates as you work.

Each kind also has a scope — to one project, or universally. Two questions, two answers each:

Operational — how to actContextual — what & where
Project Semantic & procedural memory Episodic memory + the project binder
Universal Global rules + CLAUDE.md The Library
A third pillar: the structural blueprint

The matrix above holds the knowledge. One more durable surface frames the work itself: the plan tree — spec, manifest, tasks, handoff. It's neither operational nor contextual; it's the project's blueprint, authored by you and the system together. So the honest picture is three pillars — operational, contextual, and structural — not a tidy 2×2.

"Context" is the filing cabinet, not the desk

The word is overloaded, so be precise: context here means the durable material on disk that frames the work and persists across sessions — the filing cabinet. That is not the assistant's working window, the finite attention it holds during one live conversation — the desk. Durable context is loaded onto the desk at session start; the desk is wiped clean when the session ends. Everything on this page is about the cabinet.

How it loads — small always-on, large on-demand

The working window is finite, so access is tiered by cost. Nothing that loads automatically is large; the heavy reference and the full project record sit one deliberate step away.

TierWhat
Always-onCLAUDE.md, rules, the small project-memory index, the episodic pointer, the project's situating card (_situating.md), and the active plan's goal anchor (its spec head)
On-demandthe binder cover and indexes, a Library article, a plan's detail — read when the work reaches for it
Never auto-loadedthe Workshop scratch bench — read only when research is promoted

The two insights that make this one story

  • "Memory" is not one thing — it straddles the line. Of memory's three kinds, semantic and procedural are operational (they govern behavior), while episodic is contextual (it records what happened). The project binder then continues that contextual job outside the formal memory structure — episodic memory and the binder doing the same work, one inside the memory system and one beside it.
  • Project state stacks at three granularities. To answer "where do we sit?", three contextual surfaces work coarse-to-fine: episodic memory gives the high-level "what we did last session"; the project binder holds the mid-level roll-up; and claude-mem, if installed, supplies the granular observations.
Why this split — and why it survives

The load-bearing principle: auto-generated and user-authored surfaces survive; hand-maintained status rots. Operational surfaces last because you own and update them; contextual surfaces last because the system derives them. A hand-maintained "project status" file would rot the moment attention slipped. The three-kind memory split also converges with independent memory science (the episodic/semantic and declarative/procedural distinctions) — convergent, not invented.
