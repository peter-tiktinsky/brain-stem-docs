Sessions · keeping your place

Checkpoints

A long session fills its working window, the platform compacts the history into a summary, and the verbatim detail is gone. A checkpoint is the autosave for your train of thought: a small structured file that captures where you are, so a compacted session can pick up exactly where it left off.

One file per session: ~/.local/state/brain-stem/sessions/<session-id>/checkpoint.md.

Context pressure — the fuel gauge

Context pressure is just how full the working window is, as a percent. The fuller the tank, the more urgently the system insists you save your place. It's a ladder, not a single alarm:

~35% · autosavea lightweight checkpoint is written silently, if none is fresh
›
~45% · gentle nudge"checkpoint at your next natural break"
›
~48% · mandate"checkpoint now before responding" — re-fires every message until fresh
›
~48–90% · stop-gaterefuses to let the session end on a stale or missing checkpoint
The stop-gate, precisely → in the ~48–80% band the checkpoint must be fresh (<10 min) to stop; in ~80–90% it must merely exist; above ~90% a safety valve always lets you stop (the window's too full to keep working anyway).

What a checkpoint holds — the Continuity Block

The checkpoint follows a fixed schema, the Session Continuity Block. Nothing is silently skipped — a field that can't be filled is marked [MISSING] rather than dropped. The first three fields are written as plain single lines on purpose: the pre-compaction hook counts them to recognize a freshly written rich checkpoint and avoid clobbering it. 

# Session Continuity Block
plan_id:        the active plan
phase:          where in it
task_id:        current task + status
## completed_steps     what's done this session
## files_modified      absolute paths touched
## key_decisions       choices made + why
## next_steps          the ordered resume
## ac_status           acceptance-criteria checklist
## current_blocker     the live blocker, if any
context_pct_at_checkpoint:   the gauge at write time

Three writers, one file, freshest wins

Three things can write the checkpoint, and they have a precedence so the richest snapshot always survives — all target the same file:

WriterWhenRichness
/session-checkpoint skillOn request, or auto-fired by the nudge/mandateRichest — the full block
The per-message hook~35% autosave, only if none is freshLightweight
The pre-compaction hookJust before compaction, only if none is freshMechanical fallback — no model call
Why save and restore are split across two moments

The save must happen at compaction — that's the last moment before detail is trimmed. But the restore can't: the platform doesn't allow a script to inject context back into the conversation at the compaction moment. So restore happens at the next session start — the harness restarts the session marked "compact," the start hook reads the checkpoint back in verbatim, and rotates the old one to a dated archive (a move, not a delete, so prior state is kept as history). It's the write-ahead-log recovery pattern, forced by which moments may inject context.

That restore is about this work thread — which plan, which task. Separately, a second hook runs at every session start (not only after a compaction) and loads a short project-orientation card: the plan roster, the project's overall status, the active focus, and the latest handoff headline — so the session also knows what project it is in and where that project stands. The two answer different questions; the card lives with the project binder.

Reference

  • Checkpoint file — ~/.local/state/brain-stem/sessions/<session-id>/checkpoint.md (one per session).
  • Writers — the /session-checkpoint skill (richest; atomic write + verify; failure mode is block-and-log), prompt-context.sh (~35% autosave + nudges), pre-compact-checkpoint.sh (fallback extract).
  • Stop-gate — stop-checkpoint-check.sh uses exit code 2 to keep a session going until the checkpoint is fresh.
  • Thresholds — the warn/mandate percents are settings-driven; the stop-gate bands are fixed in hook code (the R-26 session-continuity rule set).
  • Project orientation at session start — session-start-project-context.sh (the newest SessionStart hook) loads the project's situating card (_situating.md) as context before the first turn, on every session start; advisory and fails open. Distinct from the checkpoint: the checkpoint carries this work thread's state, the card carries project-wide orientation. See The project binder for the card's content and lifecycle.
