How a project's work is organized

Project organization

brain-stem organizes everything you do into two work surfaces, split by what kind of work it is. Coding work lives in code repositories, where Claude Code already feels at home. Non-coding work — strategy, consulting, writing, planning, and the deliverables they produce — needs its own durable, Claude-readable home. This page frames both, then goes deep on the second, because the second is the one brain-stem had to build.

The non-coding surface is surfaced in the vault as Work/ · file type deliverable.

Coding and non-coding work surfaces

Both surfaces get the full brain-stem treatment — memory, the project knowledge base, plans, the library, governance. The only things that differ are where you launch and where the finished artifact lives. That single split decides which surface you're working on.

Coding work surface

The code repository

You launch from a code repo and Claude reads and writes the code as it always has. brain-stem layers the project knowledge base over it; nothing about the code's home changes.

  • Launch from a code repo — ~/Code/widgetlib
  • The artifact is the repository (the code), in git
  • Owned by Claude Code natively — nothing new needed
  • brain-stem adds the project knowledge base on top
Non-coding work surface

The Work surface

There's no repository to anchor the work, so the Work surface is that anchor: a durable, governed, Claude-readable home for deliverables. Everything below is about this surface.

  • Launch from a Work-folder project — ~/work/acme-rebrand
  • The artifact is deliverables in the Work folder
  • Owned by the Work surface (the rest of this page)
  • brain-stem adds the project knowledge base and the deliverable structure
Coding work, briefly

The coding surface is the self-explanatory one within Claude Code: you launch in a repository, Claude reads and writes the code, and brain-stem layers the project knowledge base (binder, memory, library) over it. The code's home doesn't change, so it needs no further explanation here. The rest of this page is the non-coding Work surface, which is where brain-stem had to build something new.

The Work folder is a master directory

~/work/ is not itself a project — it's a master directory that holds one folder per non-coding project. Each of those folders is a launch directory: you cd into it and start Claude there, and that folder becomes the project (see What loads each session). One Work area, many projects: 

~/work/
├── acme-rebrand/          ← a consulting engagement (launch Claude here)
├── q3-board-deck/         ← a writing project
└── market-entry-study/    ← a research project

What ships natively with the surface

Inside each project folder, the surface ships a minimal, governed starting structure: a thin CLAUDE.md (the project's identity, a pointer to its binder, and an auto-kept map of what's in the folder — which you don't hand-edit), a README.md, an append-only updates.md, and two folders — deliverables/ for polished, shareable work and reference/ for the raw material behind it (kept out of the vault's search). The exact scaffold, flat versus master and what each file is for, is laid out in the guidebook below.

A markdown file under Work/ that declares type: deliverable activates a deliberately thin, lifecycle-only contract: six required frontmatter fields (type, tags, updated, project, status [draft·delivered·superseded], audience [internal·external]) and no body structure at all — because a brief, a memo, a deck source, and a multi-page report share no common shape. Governance follows the vault's split: in a registered Work project the six required fields' presence is checked at the write, while their enum values, body structure, and cross-file parity are reconciled by a scoped librarian pass at session close; an unregistered folder is audit-time only until you register it.

The base structure is a floor, not a ceiling — you add whatever folders the project needs and Claude reads whatever structure it finds. Setting one up, step by step, with worked flat and multi-workstream examples: The work-folder guidebook.

Either surface rolls up into the project binder

Here is the part both surfaces share. Whichever one you launched from, a session's durable output — its research, its decisions, its handoff (and, for non-coding work, the deliverables it produced) — rolls up into the same place: the project binder, anchored by the project's hub.md home page. The artifact differs (a repository vs. deliverables); the knowledge base does not.

A session — whichever surface you launched from
Coding projectresearch · decisions · a handoff — the code itself lives in git
Non-coding projectresearch · decisions · a handoff — plus the deliverables it produced
▲ Regardless of type, each session's durable work files up into the same binder
The project binder — one roll-up per project, across all its sessions
hub.md — the project's home pageindexes the research, decisions, and handoffs — and, for non-coding work, the deliverables, joined on the shared project: field. Plans finish and close; the deliverables stay live under the project.

A deliverable is independent of any single plan but never orphaned: it joins its project through hub.md's Deliverables pointer, keyed on project:, so it survives every plan of that project.

For the binder itself — its cover, the generated research index, decision log, and handoff chronicle, and how a project is identified — see The project binder.

Where the two surfaces interplay

The surfaces are separate, but they can feed each other. The common case: you run a consulting engagement on the Work surface and want a live project-management dashboard for it — a coding project. Because a vault-writer's pipeline produces a structured layer (JSON process files and a SQL ledger) before it ever writes markdown, the dashboard can read that feed directly and deterministically, while the Work-folder markdown stays the curated human source of truth. That composition — and the writer pipeline behind it — is covered in full under The vault-writer system.

Reference

  • ~/work/<project>/ — the per-project launch directory, surfaced in the vault as Work/<project>/; ships CLAUDE.md, deliverables/, reference/, README.md, updates.md. The CLAUDE.md map block is regenerated by librarian work-map-generate.
  • governance/file-type-contracts/deliverable.md.json — the thin lifecycle contract (status & audience enums; no body governance).
  • governance/frontmatter-rules.json#types.deliverable — the six required frontmatter fields.
  • templates/hub-template.md — the project home page; its Deliverables pointer is the join, keyed on project:; the binder is where either surface's work rolls up.
  • A vault-writer — an automated, registered system that feeds structured content into a project's Work area; its final-layer SQL/JSON output is the deterministic feed a coding dashboard can consume (see The vault-writer system).
