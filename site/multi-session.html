Sessions · staying out of each other's way

Multi-session coordination

You can have several sessions open in one project at once — and without coordination they'd quietly overwrite each other's work. brain-stem keeps a shared whiteboard the open sessions read to stay clear of one another, and surfaces a warning the moment two sessions reach for the same file.

A machine-local registry — bookkeeping for running programs, deliberately not in the vault.

The session registry

Every open session registers itself in one ephemeral file. Think of it as a whiteboard the sessions read to stay out of each other's way. Each row carries what another session needs to know:

FieldWhat it records
pidThe long-lived Claude process (resolved by walking up from the hook, not the transient subshell).
statusactive · closed · closed-pending-reconciliation.
last_heartbeatA liveness timestamp, so a dead session can be told from a live one.
touched_filesThe files this session has edited — populated after each write.

Writes to the registry are lock-guarded and atomic (temp file, then rename), so two sessions updating it at once never corrupt it. It lives in machine-local state, not the vault — it's bookkeeping for the running programs, not knowledge.

Peer awareness and the same-file warning

On each message you send, the per-message hook reads the registry and speaks what the other open sessions are doing into the conversation:

  • a short summary of the other active sessions;
  • the load-bearing one — a warning when a peer has touched the same file you've also touched: "Overlapping files with peer sessions: … Re-read before editing";
  • a note of who left work pending, and a recommended close mode when you mention finishing.
Why a per-message hook does this

The registry's peer state changes between every message — sessions open and close while you work — so it has to be read at message time, not baked in once at the start. The message-submit hook fires on every message you send and hands what it reads into that turn; that is why the registry feeds it. (A separate session-start hook injects pre-baked project context before your first turn, but only the per-message hook can carry state that changes mid-session.) The registry collects the facts; the message-submit hook speaks them.

Last session out does the cleanup

The status field powers a simple discipline for closing cleanly. When you end a session, it marks its own row closed. If peers are still active, your close does only your own part and defers the shared cleanup to whoever closes last. The reconciler — which reaps stale, closed, and pending rows under a lock — is idempotent, so running it twice changes nothing. Two closing sessions never trample each other or run the global cleanup twice.

That deferral is the bridge to the next page: closing a session is itself a small reconciliation, and a full close runs a whole cleanup chain. (See Session close.)

Reference

  • The registry — ~/.local/state/brain-stem/.coordination/session-registry.json; lock-guarded, atomic writes; machine-local (never in the vault).
  • Peer awareness — prompt-context.sh reads the registry on each message and emits the summary + same-file overlap warning.
  • Touched-file tracking — a post-write hook appends each edited path to the session's touched_files.
  • Reconciliation — reconcile-sessions.sh reaps stale/closed/pending rows under a lock; idempotent.
