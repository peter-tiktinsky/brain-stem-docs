Plans & orchestration

Plans & orchestration

A "plan" is a project run as a tracked, multi-step thing on disk: a folder per project, a fixed goal, a task list with dependencies, a lifecycle status, and — when you want it — a runner that executes the tasks with a human in the loop. This is the odd-one-out pillar: Claude Code never forces you to run a project this way, so the whole structure is an opt-in methodology rather than a necessity — the one honest asterisk. But once you adopt it, the shape is not a matter of taste; every rule lands on the answer the field reached independently, which the repo calls convergence-grade. The design exists to remove two failure modes seen everywhere: the agent claims work is done when it isn't, and parallel work-streams tangle.

brain-stem choseoverbecauseground
Make the control file the truth; generate the to-do list from it Tracking completion in a hand-edited markdown checkbox That is exactly where real tools' bugs cluster — Kiro's agent reported a box ticked that wasn't Kiro issues, and a review of three SDD tools found checklist status is the failing part Fowler/Böckeler. One representation is definitive, the rest generated DRY/SSOT — Task Master makes the same call Task Master. convergence
Machine-stamp "verified"; treat "completed" as a claim; refuse "closed" without "verified" Letting a person mark a project done "Done" must be unforgeable: a status you can set without a check is a claim, and a deterministic hook acts on it regardless of whether the work landed spec-gaming. The verified-vs-accepted line is standard project management PMI/PMBOK. convergence
Forbid sub-plan peers from talking — everything routes through the master An agent-teams model where teammates message each other directly Free all-to-all coordination recreates the tangled web that is a leading cause of multi-agent failure — ~37% of failures are inter-agent-misalignment (coordination) failures MAST. The master/sub flow maps onto the orchestrator-worker pattern: workers report up only CC sub-agents. constraint
A human-gated runner that proposes and waits A fully autonomous, unattended daemon No task fires without a human reasoning or approval point — irreversible tasks halt for approval (human-in-the-loop), reversible ones proceed under supervision. An unsupervised runner inherits documented multi-agent failure rates MAST; it is built on the platform's headless mode headless mode. convergence
Keep decisions as first-class, immutable, supersession-tracked records Burying decisions in spec prose, or adding dedicated ADR tooling Recording the rejected options is what makes a record valuable later Nygard ADR MADR; the two-tier split is the Spec-Kit constitution pattern Spec-Kit and counters the context-decay failure SDD names Thoughtworks. A sub-120-line ADR is already what Google's ReasoningBank reinvents. convergence
Capture ideas cheaply; mint a project number only at graduation Minting a project number for every stray idea A number per idea burns a permanent slot on each of dozens of discarded ones and makes the numbered space meaningless. The four-file quartet maps onto and exceeds the SDD tools AWS Kiro Spec-Kit with a machine-parseable manifest and write-time enforcement. design judgment
Enforce through Claude Code's own hooks, in three layers Prose standing-instructions, a single write-time blocker, or an async-only reconciler Hooks fire regardless of whether the model cooperates; prose decays over sessions #33603 #56393. A single blocker can't see cross-file invariants; an async-only sweep leaves a spec-gaming window. Every write goes through one doorman — complete mediation Saltzer & Schroeder. convergence
An append-only handoff journal that carries the resume pointer Relying on the assistant to remember its state across a long session The context window is finite and compaction trims in-the-moment detail away CC context-window, so durable state must live on disk where a post-compaction session cold-starts off the top block. constraint
Ship dispatch governance as a minimum-viable safe envelope first Shipping all six dispatch surfaces at once Three surfaces (pre-dispatch scoping, brief-lint, circuit-breaker) are the minimum safe envelope; budget caps, a standalone verifier, and mechanical retry are gated to ship once post-release telemetry confirms the envelope holds. The independent human gate is in scope from v1.0.0. design judgment

Project binder

Project binder & context

Three decisions from the context & library pillar belong here because they govern how the project binder is maintained and how project context reaches a session. All three serve the same goal: keeping project state correct with no manual upkeep. The load-bearing thread is the same one that runs through the whole context pillar — auto-generated surfaces survive while hand-maintained ones rot.

brain-stem choseoverbecauseground
The project binder is generated, never hand-written; membership is automatic A hand-maintained "project status" file and membership roster A status file kept current by a person rots the moment attention moves on. Deriving the binder from the project's plans keeps it correct with no upkeep — the central rot mode the system exists to kill, mirroring PARA's active-binder split PARA. convergence
A generated, automatically-loaded situating card (_situating.md) as the eager project-orientation surface Relying on hub.md as the eager session-start surface — a file nothing automatically writes, so every fresh project would begin with a guaranteed read-miss The session-start context channel is the only way to inject project orientation before the model's first turn without user action. The card is regenerated atomically every session from machine-derivable fields alone — plan roster, per-plan status, active focus, latest handoff headline, pointers — so there is nothing hand-maintained to rot, and it is kept small so it never competes for the context window. hub.md stays as the curated, read-on-demand cover, minted at registration. constraint convergence
Project identity is the launch directory, with no declared roster A declared membership list A hand-maintained roster drifts the moment you forget to update it; reading identity from where you launched means nothing to keep in sync. Johnny.Decimal is flagged an anti-pattern precisely because fixed addresses fight auto-membership binders Johnny.Decimal. convergence
